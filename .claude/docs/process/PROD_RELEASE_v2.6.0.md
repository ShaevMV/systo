# Runbook: выкат v2.6.0 на прод + миграция данных заказов

> **Зачем:** v2.6.0 сменил формат `order_tickets.guests` на **per-guest** (`ticket_type_id`/`promo_code`/`price_snapshot` на каждого гостя) — это **BREAKING**. Старые прод-заказы в legacy-формате **упадут при чтении** новым кодом, пока не прогнана разовая миграция `order:migrate-to-guests-format`. Обычная `php artisan migrate` её НЕ запускает.
>
> **Стратегия (решение владельца 2026-06-20):** сперва прогоняем на **стейдже на копии прод-базы** (доказываем на реальных данных), затем — прод. Миграция вшита в CI/CD стейджа (идемпотентный шаг после `migrate`).

---

## Свойства команды миграции (проверено)

`php artisan order:migrate-to-guests-format`
- **По умолчанию dry-run** — без изменений в БД, только отчёт.
- `--apply` — реально пишет; **перед записью делает inline-бэкап** таблицы `order_tickets_backup_v2_6_0_<timestamp>`.
- `--no-backup` — пропустить inline-бэкап (для повторных/идемпотентных прогонов).
- **Идемпотентна:** повторный прогон пропускает уже мигрированные заказы (детект по `price_snapshot` у первого гостя).
- Unit-тест `OrderGuestsMigratorTest` — **13/13 зелёный**.

## Этап 1 — репетиция на стейдже (на копии прод-базы)

> Цель: прогнать миграцию на **реальных** прод-данных в безопасной среде до прода.

⚠️ **ПДн (152-ФЗ):** прод-дамп содержит реальные персональные данные гостей. Стейдж менее защищён (открыт phpMyAdmin за basic-auth). Минимизация: ограниченный доступ, **удалить прод-данные со стейджа после репетиции** (вернуть тестовый сидер), не оставлять дамп на диске.

1. **Снять свежий дамп прода** (только `systo`):
   ```bash
   # на прод-сервере
   mysqldump -u root -p systo > systo_prod_$(date +%F).sql
   ```
2. **Импортировать в стейдж** (контейнер `mysql-staging`):
   ```bash
   # залить файл на стейдж-сервер, затем:
   ssh deploy@77.222.32.244 'docker exec -i mysql-staging sh -c "mysql -uroot -p\"$MYSQL_ROOT_PASSWORD\" systo" < systo_prod_YYYY-MM-DD.sql'
   ```
   Авто-сидер деплоя при этом НЕ сработает (он льёт сиды только при `festivals=0`; в прод-дампе фестивали есть).
3. **Запустить деплой стейджа** (push в ветку `staging` или `gh workflow run deploy-staging.yml`). Шаг **«Run v2.6.0 data migration»** прогонит: dry-run (план в логе Actions) → `--apply --no-backup`.
4. **Проверка** (что миграция отработала на реальных данных):
   - в логе Actions у шага миграции — сколько заказов мигрировано, без ошибок;
   - повторный прогон даёт **0 к миграции** (идемпотентность):
     `docker exec php-staging php artisan order:migrate-to-guests-format` → «0 к миграции»;
   - спот-чек в админке: старые заказы **открываются и читаются** (`order/getItem`, список заказов) без ошибок.
5. **После репетиции** — вернуть стейдж на тестовые данные (удалить прод-ПДн), напр. `gh workflow run deploy-staging.yml -f seed=fresh`.

## Этап 2 — прод (вручную, окно простоя допустимо)

> Прод-CD ещё нет (план v2.9.0) → деплой ручной. Простой допустим (правило проекта).

1. **Бэкап прод-БД** (несколько копий, вне сервера): `mysqldump -u root -p systo > backup_pre_v2_6_0_$(date +%F_%H%M).sql`.
2. **Maintenance:** `php artisan down`.
3. **Деплой кода v2.6.0** на прод (текущий ручной процесс).
4. **Dry-run миграции:** `php artisan order:migrate-to-guests-format` → проверить отчёт (число заказов, отсутствие ошибок).
5. **Применить с бэкапом:** `php artisan order:migrate-to-guests-format --apply` (создаст `order_tickets_backup_v2_6_0_*`).
6. **Проверка:** повторный dry-run = 0 к миграции; спот-чек заказов в админке.
7. **Maintenance off:** `php artisan up`.

**Откат (если миграция сломала данные):** восстановить из `order_tickets_backup_v2_6_0_<timestamp>` (или из дампа шага 1) + откатить код на прежний тег.
