# Общая картина системы Systo

> Документ для аналитика и архитектора. Цель — единая карта систем, потоков данных и доменных границ, чтобы принять решение, где разместить админ-панель (в частности, управление сменами/входом Baza) и как разграничить доменные модели. Решение остаётся за читателем — здесь только факты и развилки.
>
> Обновление 2026-06-20: между Backend и Baza появились сетевые каналы (ingest-API записи билета Ф3, вебхук «билет прошёл» Ф4 — оба пока default OFF), а на КПП — offline-first PWA `BazaFront` (Ф5). Карта систем перестала быть «только прямая запись в БД + sneakernet».

## 1. Карта систем

Монорепо Systo физически содержит несколько приложений, плюс внешние сервисы. Стрелки показывают направление потока данных.

```
                            ┌─────────────────────────────────────┐
                            │            ГОСТЬ (покупатель)         │
                            └───────────────┬─────────────────────┘
                                            │ покупка билета (коммерция)
                                            ▼
                           ┌──────────────────────────────────┐
                           │   qr.spaceofjoy.ru (внешний)      │
                           │   Python, отдельный репозиторий   │

                           │   биллинг-UX (МАСТЕР ЦЕНЫ)        │
                           └───────┬───────────────────┬──────┘
                                   │                   │
```

Ключевые свойства связей:

- **qr → Backend** — единственный санкционированный сетевой канал приёма заказов (S2S, `X-QR-Token`). UUID заказа qr равен UUID заказа org (без таблицы маппинга).
- **Backend → Baza** — **два пути записи билета.** (1) Приоритетный канал — **ingest-API Baza** `POST /api/baza/ingest/ticket` (S2S, заголовок `X-Baza-Token`, middleware `baza.ingest`, Ф3). (2) Fallback — **прямая запись в чужую БД** через второе Laravel-соединение `mysqlBaza`. `DeliverTicketToBazaJob` сперва пробует ingest-API (`BazaIngestClient`), при сбое/неприменении пишет напрямую. **Канал по умолчанию OFF** (`BAZA_INGEST_URL`/`BAZA_INGEST_TOKENS` пусты) — поэтому прод-поведение пока прежнее (прямая запись). `routes/api.php` Baza содержит `/user` и `/baza/ingest/ticket`; `/scan` и `/enter` перенесены в `routes/web.php` под `middleware('auth')` (сессия + CSRF).
- **Baza → Backend (вебхук «билет прошёл», Ф4)** — `POST /api/v1/baza/ticketEntered` (S2S, заголовок `X-Baza-Token`, middleware `baza.webhook`, конфиг `services.baza_webhook`). Пишет факт входа в `domain_history` (`aggregate_type=ticket`, `actor_type=baza`), идемпотентно по `event_id`. **Канал по умолчанию OFF.**
- **Baza → ноутбук-сканер** — перенос данных файлом-архивом (sneakernet), без сети.
- **КПП-клиенты Baza** — старый Blade-интерфейс (`vhod.*`) + новый **BazaFront** (offline-first PWA, под `/baza/` на `vhod.*`, Ф5). См. §5.
- **FrontEnd и AdminFront** — два фронта поверх одного Backend API (JWT). FrontEnd — старый, держит прод; AdminFront — новый, на `/admin/`.

## 2. Роли систем в гибридной модели

Архитектура — **гибридная** (зафиксировано в памяти проекта, разворот org → admin-only): коммерция отделена от исполнения.

| Система | Роль в модели | Чем владеет (мастер) | Чего НЕ делает |
|---------|---------------|----------------------|----------------|
| **qr.spaceofjoy.ru** | **Коммерция** | Витрина, оформление заказа, промокод как UX, биллинг, **цена заказа** (мастер цены) | Не генерирует билеты/PDF, не пишет в Baza, не сканирует |
| **Backend (org)** | **Каталог + билеты + рассылки** | Каталог фестивалей/типов билетов, сами билеты, PDF/QR, анкеты, письма (EmailDelivery), журнал доставки в Baza (BazaDelivery) | Не считает цену (приходит готовой из qr-контракта), не контролирует физический вход |
| **Baza** | **Вход / сканирование** | Факт прохода гостя на КПП (`change_id`/`date_change`), смены персонала, счётчики прошедших | Не создаёт билеты (получает их записью из Backend), не знает цен/денег |

Важные следствия для архитектора:

- **Цена** — мастер на qr; Backend/org промокод к цене не применяет, `total_price` приходит готовым. Промокод в org-заказах хранится только для справки.
- **Каталог** (фестивали/типы/опции/цены) остаётся **мастером на org**, несмотря на переезд коммерции.
- **Baza денег не знает вообще** — в её схеме нет ни одного финансового поля (см. §4 и §5).

## 3. Потоки данных по сценариям

### (а) Покупка → выдача → запись в Baza → сканирование на входе

```
1. Гость покупает на qr.spaceofjoy.ru (или на старом FrontEnd через order/create).
2. qr считает цену (мастер цены) → S2S POST /api/v1/qrOrder/create (X-QR-Token).
   Заказ ложится в qr_orders org. uuid qr == uuid org.
3. Backend запускает выдачу (пайплайн qr-заказа ИЛИ классический org-флоу при PAID):
     • CreateTickets — создаёт билеты
     • ProcessCreatingQRCode — PDF (DomPDF) + QR (PNG с логотипом).
       В QR зашита ССЫЛКА org: /newTickets/<uuid> (электронный) либо
       https://org.spaceofjoy.ru/ticket/live/<шифр> (живой, AES-256-GCM, KEY_LIVE_TICKET)
     • SendOrderEmail — письмо с PDF (через EmailDelivery, см. поток «в»)
     • PushToBazaStep / PushTicketsCommandHandler — ставит доставку билета в Baza
4. ДОСТАВКА В BAZA (модуль BazaDelivery, асинхронно + трекинг):
     BazaDeliveryDispatcher → запись baza_deliveries(queued) → DeliverTicketToBazaJob
     Job по target пишет ПРЯМО в БД Baza (соединение mysqlBaza):
       el_tickets / spisok_tickets / live_tickets / auto
     Машина: queued → sending → delivered / failed. Ретрай backoff [30,120,600],
     КАП 10 попыток. Сбой Baza НЕ роняет выдачу билета (билет/письмо уже у гостя).
     Каждая попытка → событие в domain_history (aggregate_type=baza_delivery).
5. Гость приходит на КПП. Персонал Baza сканирует QR камерой (/scan → /api/scan):
     DefineService по URL определяет тип, находит строку билета в БД Baza.
6. ВПУСК (/api/enter): находим открытую смену билетёра (change_id),
     +1 в счётчик смены нужного типа, билету ставим change_id + date_change=now.
     Заполненный date_change = «уже проходил» (защита от повторного входа — на фронте).
```

Идентификаторы сквозные, маппинг не нужен: `el_tickets.uuid` = id билета org, `spisok_tickets.ticket_uuid` и `live_tickets.el_ticket_id` — ссылки на UUID org, `auto.order_id` — UUID заказа-списка, `festival_id` — общий UUID фестиваля.

### (б) Приём qr-заказа

```
qr.spaceofjoy.ru ──S2S POST /api/v1/qrOrder/create (X-QR-Token)──► Backend
                                                                      │
  идемпотентно по id (повтор того же uuid не создаёт дубль)          │
  Поддержано ДВА режима жизненного цикла:                            ▼
   (1) Двухшаговый: create со status=«создан» → org шлёт         qr_orders
       письмо «заказ создан», билеты НЕ выпускает; затем
       qrOrder/changeStatus/{id} → «оплачен» → выдача билетов.
   (2) Одношаговый: create сразу со status=«оплачен» → немедленная выдача.
  Выдача — один раз, защита по issued_at (повторный «оплачен» билеты не выдаёт).
  changeStatus НЕ удалён. В domain_history всё пишется actor_type=qr.
  Admin-просмотр (read-only, ПДн): qrOrder/getList|getItem|getHistory|getPipeline.
```

### (в) Доставка письма (модуль EmailDelivery)

```
Событие (order_paid / list_approved / user_registered / ...) или
S2S-приём письма от витрины (POST /api/v1/emailNotification/send, X-QR-Token)
        │
        ▼
MailDispatcher.send(EmailEvent, EmailContext, Mailable)
   • создаёт email_messages(queued) + сериализует Mailable в колонку
   • slug шаблона выбирает TemplateBindingResolver (оси: types_of_payment >
     event > ticket_type > order_type > festival), fallback на blade
   • SendEmailJob (tries=3, backoff [30,120,600]):
        queued → sending → sent (передано на SMTP) → [delivered → opened] / failed
        delivered/bounced требуют провайдера с вебхуками (AF-6, ещё нет)
        opened — пиксель GET /api/v1/mail/open/{token}.gif (за флагом, 152-ФЗ)
        │
        ▼
     SMTP / Mailpit (на staging). Трекинг виден в админке «Доставка писем».
     source ∈ qr_pipeline / qr_intake / org_event.
```

### (г) Офлайн-смена на КПП

```
ПЕРЕД фестивалём (есть интернет):
  admin → /sync → «Выгрузить ZIP» (ExportApplication / artisan baza:export)
  6 таблиц (auto, changes, el_tickets, live_tickets, parking_tickets,
  spisok_tickets) → построчно в NDJSON (.jsonl) → один ZIP.
        │ перенос файлом (флешка / локальная сеть) — sneakernet
        ▼
НА КПП (часто без интернета, ноутбук-сканер, офлайн-копия Baza):
  admin → /sync → «Загрузить ZIP» (ImportApplication / baza:import)
  UPSERT по id + сравнение updated_at (новее → UPDATE, иначе SKIP).
  Удалённые на источнике НЕ трогаются. Per-row try/catch.
        │
  Билетёры сканируют офлайн, ставят отметки входа (change_id/date_change).
        │ обратный перенос ZIP
        ▼
ПОСЛЕ (или между точками входа):
  выгрузка отметок входа обратно на сервер тем же ZIP (слияние по updated_at).
```

> Статус офлайна: код export/import есть и симметричен. Но автономный клиент (локальный SSL/mkcert для камеры, offline docker-compose bundle) — **пока только план v2.8.0**, кода в репозитории нет. На прошлом фестивале сканер не заработал из-за отсутствия HTTPS в локальной сети без интернета. При нескольких офлайн-ноутбуках слияние идёт по `updated_at` (последняя строка побеждает, без мержа полей).
>
> Корректировки владельца (2026-06-20, `.claude/specs/baza-f5-pwa.md`): локальный узел-ноутбук на КПП **опционален** — его может не быть. Истина — облако (боевой сервер); телефоны (BazaFront PWA) синкаются с облаком **напрямую по сотовой связи** оппортунистически, а не через sneakernet-ZIP. На **полный блэкаут** (ни сотовой, ни Wi-Fi, ни узла) консистентность держит **физический браслет** (цвет = тип билета, Ф6): получил браслет — визуально уже впущен, цифровые журналы сверяются позже («первый впуск побеждает», append-only журнал `baza_entry_events`, PR-8). Смена — 10 часов.

### (д) Поиск гостя без QR

```
При ingest-доставке (Ф3) org прикладывает богатый блок search
(хранится в baza_deliveries.search_blob, миграция 2026_06_20_160000):
  ФИО / телефон / telegram / госномер авто / имя ребёнка / № заказа.
DeliverTicketToBazaJob прокидывает его в ingest-API → Baza индексирует
в таблицу ticket_search (миграция 2026_06_20_150000).
        │
        ▼
На КПП (/search) — секция «Найдено по данным (поиск без QR)»:
персонал ищет гостя по любому из этих полей, когда QR не работает
(частый случай). Наполняется ТОЛЬКО при включённом ingest-канале (Ф3);
при прямой записи mysqlBaza индекс не пополняется (PR #113/#114).
```

## 4. Доменные модели и их границы (Bounded Contexts)

Это ключевой раздел для разграничения. В системе есть два чётко разных домена и зона их пересечения.

### Домен A — «Заказ и билет» (Backend / org)

- **Сущности:** `OrderTicket` (AggregateRoot), `Ticket`, каталог (`Festival`, `TicketType`, `Option`, `Location`), `PromoCode`, `Questionnaire`, плюс трекинг-модули `EmailDelivery` и `BazaDelivery`.
- **Язык домена:** заказ, статус заказа (матрица переходов), цена/скидка (но цена — из qr), PDF, QR, анкета, рассылка.
- **Идентификатор:** UUID (Ramsey).
- **Ответственность:** довести заказ от приёма до выданного билета (PDF/QR/письмо) и доставить билет в систему входа.
- **Где заканчивается:** на факте «билет создан и записан в Baza». Дальше судьбу билета на входе Backend не отслеживает (кроме статуса доставки в `baza_deliveries`).

### Домен B — «Вход / контроль доступа» (Baza)

- **Сущности:** `el_tickets`/`spisok_tickets`/`live_tickets`/`parking_tickets`/`friendly_tickets`/`auto` (проекции билетов для сканирования), `changes` (смена персонала), `users` (персонал КПП).
- **Язык домена:** билет на входе, kilter (номер для сканирования), проход (`date_change`), смена дежурства, счётчики прошедших, браслет (цвет по типу).
- **Идентификатор:** числовой `id` (auto-increment) для своих сущностей; UUID — только как **внешняя ссылка** на org-билет.
- **Ответственность:** опознать билет на КПП, решить «пускать/не пускать», отметить факт прохода, вести смены персонала.
- **Где начинается:** в момент, когда строка билета появилась в БД Baza (её записал Backend) и персонал начал сканировать.

### Зона пересечения (shared)

| Что пересекается | Как реализовано | Риск/особенность |
|------------------|-----------------|------------------|
| **Билет существует и там, и там** | Backend пишет строку в таблицы Baza напрямую (соединение `mysqlBaza`). Контракт неявный — `TicketResponse::toArrayForBaza()` / `toArrayForSpisok()` white-list колонок | Нет общего DTO/схемы/версии. Схему таблиц владеет Baza (миграции Baza), наполняет Backend. Изменение схемы Baza → молчаливый сбой записи из Backend → ретрай → терминальный `failed` после 10 попыток |
| **Идентификаторы** | `uuid`/`ticket_uuid`/`el_ticket_id`/`order_id`/`festival_id` — общие сквозные значения | Согласованность держится вручную, единого справочника соответствия нет |
| **Статус билета** | `status` (paid/cancel/...) приходит из org; Baza его только читает для «пускать/нет». Отмена → org ставит `status=cancel` в Baza | Фронт-сканер допускает статусы `paid_for_live`/`approve_list`, которых нет в Baza-версии `Status` VO — частичный рассинхрон статусов |
| **Шифрование живого билета** | Общий секрет `KEY_LIVE_TICKET`: org шифрует номер в QR, Baza расшифровывает | Общий секрет между двумя приложениями |
| **Одна СУБД** | На staging/prod Backend и Baza ходят в один MySQL-инстанс (разные схемы) | Нет изоляции отказов БД между коммерцией и входом |

**Вывод по границам:** домен «вход/смена/впуск» (Baza) **самодостаточен и слабо связан** с доменом «заказ/билет» (Backend) — связь только через проекцию билета в БД и сквозные UUID. Это естественный шов (seam) для любого решения о размещении управления.

## 5. Вопрос размещения админ-панели

### Какие админки уже существуют

| Админка | Стек | Где живёт | Домен / что покрывает | Auth |
|---------|------|-----------|------------------------|------|
| **FrontEnd (старый)** | Vue CLI / webpack, Bootstrap | `org.spaceofjoy.ru` (`/`) | Публичная витрина + **текущая** админка org: заказы, типы билетов, промокоды, анкеты, локации | JWT (Backend) |
| **AdminFront (новый)** | Vite + PrimeVue Sakai | `/admin/` (на staging — мини-cutover) | Новая **admin-only** админка org. Разделы: Управление (дашборд, анкеты), Заказы (qr/оргвзносы/дружеские/списки), Справочники (каталог), Письма (шаблоны/привязки/доставка), Билеты (доставка в baza) | JWT (Backend) |
| **Веб-интерфейс Baza** | Laravel Blade + Bootstrap 4 + jQuery (тема Black Dashboard) | `vhod.spaceofjoy.ru` | Домен **входа**: сканер `/scan`, поиск `/search`, **смены** `/report` + `/change/*`, **синхронизация** `/sync/*`, **права** `/permissions` | **Session** (web guard) + RBAC-матрица `baza_role_permissions` (5 ролей смены, middleware `permission:<действие>`, Ф2), НЕ JWT |

Админки сегодня живут в двух разных доменах аутентификации: org-админки (FrontEnd + AdminFront) — на JWT Backend с ролями (admin/seller/pusher/manager/curator); интерфейсы Baza (старый Blade + новый BazaFront PWA) — на сессиях Laravel. После Ф2 в Baza плоская бинарная роль `is_admin` заменена RBAC-матрицей `baza_role_permissions` (5 ролей смены, права через `permission:<действие>`, UI-редактор `/permissions`); `administrator` — суперроль.

### Где разместить управление сменами/входом Baza — факты и развилки

Управление сменами и впуском сейчас **внутри Baza** (`/report` под `permission:report.view`, `/change/*` под `shift.compose`/`shift.close`/`shift.remove`, `/sync/*` под `sync.manage` — RBAC-матрица Ф2, а не прежний `middleware('admin')`). Возможные варианты размещения — изложены как развилки, **без рекомендации**.

**Вариант 1 — оставить в Baza (как сейчас).**
- *За:* домен входа самодостаточен; смены физически нужны на КПП, где работает именно Baza (в т.ч. офлайн на ноутбуке); не нужен сетевой доступ к org во время фестиваля.
- *Против:* старый интерфейс на устаревшем стеке (Blade + jQuery + Bootstrap 4, тема Black Dashboard, есть мёртвые страницы темы) — параллельно идёт переезд на BazaFront PWA (Ф5); нет аудита действий со сменой; URL захардкожены на прод-домен (ломает staging/офлайн). Права уже не плоские — после Ф2 это RBAC-матрица `baza_role_permissions`.

**Вариант 2 — перенести/продублировать в новую org-админку (AdminFront).**
- *За:* единый современный UI (Vite+PrimeVue), единые роли JWT, единый дашборд; в AdminFront уже есть раздел «Доставка в baza» (но это трекинг доставки билета, а не смены/впуск).
- *Против:* смены и впуск **физически работают на КПП**, часто офлайн — org-админка (JWT, сетевой Backend) там недоступна. Дублирование потребует либо сетевого доступа к Baza, либо синхронизации данных смен в org. Два домена auth (JWT vs session) пришлось бы как-то связать.

**Вариант 3 — отдельная панель / гибрид.**
- Оперативные действия КПП (сканер, впуск, смены) — остаются в Baza (полевой инструмент, офлайн-готовый); сводная аналитика/отчёты по сменам и доставке — поднимаются в org-админку (AdminFront), читая данные из БД Baza (общий MySQL-инстанс) или через выгрузку.
- *Развилка:* где «мастер» данных смены — в Baza (создаётся на месте) или в org. По корректировкам владельца (2026-06-20, `baza-f5-pwa.md`) узел-ноутбук на КПП **опционален**, истина — облако; телефоны (BazaFront PWA) синкаются с облаком напрямую по сотовой связи. На полный блэкаут anti-double-entry обеспечивает **физический браслет** (цвет = тип билета), а не ноутбук-сканер как обязательное звено.

### Что важно учесть при любом варианте (факты из кода)

- **Финансов в Baza нет вообще.** В таблице `changes` и во всём модуле нет ни одного денежного поля (выручка, сумма, наличные/безнал, инкассация). Если админ-панель должна показывать финансы смены — данных в Baza для этого нет; источник суммы — org/qr (Baza цен билетов не знает).
- **Нет материальной ответственности / старшего смены / комментариев начальника** — соответствующих полей и сущностей нет. Состав смены — плоский JSON-массив `user_id` без ролей внутри смены.
- **Нет аудита** действий со сменой (в org для этого есть `domain_history`, в Baza аналога нет) — по-прежнему верно после Ф2, хотя роли/права теперь матричные (`baza_role_permissions`). Удаление смены — жёсткое (`delete`, без soft delete).
- **Итог отчёта неполный:** `ReportTotalDto.total = live + drug + spisok + el` — парковки и авто в общий итог не входят (отдельными полями суммируются). Зафиксировать, баг это или намеренно.
- **Технические мелочи:** `count_auto_tickets` отсутствует в `$fillable` модели `ChangesModel` (наполняется только `increment`); `festival_id` зашит константой UUID в репозиториях и `SaveChange`; `live_tickets`/`parking_tickets` не фильтруются по `festival_id` при поиске.
- **Безопасность входа (security-фикс, PR #84):** эндпоинты `/api/scan` и `/api/enter` перенесены в `routes/web.php` под `middleware('auth')` (сессия + CSRF); id сотрудника берётся из сессии (`Auth::id()`), а не из тела запроса. Защита от повторного впуска теперь **и на сервере**: `EnterTicket::skip()` бросает исключение, если билет не найден или уже впущен (заполненный `date_change` блокирует повторную отметку), а не только на фронте.
- **Контракт Backend→Baza неявный** (white-list колонок, общая СУБД, нет API) — любое разделение/перенос моделей должно решить, формализовать ли этот контракт (общая схема/версионирование либо переход на ingest-API Baza) или оставить как есть.
