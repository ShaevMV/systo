# Baza — система контроля входа на фестиваль

## 1. Что это такое (простыми словами)

**Baza** — это отдельное небольшое веб-приложение, «система входа» на фестиваль. На воротах (КПП) стоит сотрудник с телефоном или ноутбуком, наводит камеру на QR-код в билете гостя — и Baza мгновенно показывает: кто это, какой у него билет, оплачен ли он и не проходил ли уже кто-то по этому билету раньше. Если всё в порядке, сотрудник нажимает «Пропустить» — и гость заходит.

По сути это упрощённая **копия билетов** из основного приложения (Backend / org) плюс **журнал проходов**: кто, когда и в какую смену был впущен.

> ⚠️ **Важная оговорка про название.** В старой документации (файл `CLAUDE.md`) Baza названа **«Аутентификация»**. Это вводящий в заблуждение ярлык. На самом деле «аутентификация» в Baza — это лишь её собственный **логин для персонала** (вход билетёра по email/паролю), а вовсе не назначение приложения. Реальная задача Baza — **контроль входа на КПП фестиваля**: сканирование QR, впуск гостей, учёт смен. На стенде это видно даже по адресу — приложение живёт на поддомене **`vhod.*`** (`vhod` = «вход»), например `vhod.staging.spaceofjoy.ru`.

**Корректная формулировка:** Baza — это система контроля входа (КПП) на фестиваль, отдельное Laravel-приложение для персонала.

---

## 2. Зачем нужна и кто ей пользуется

Baza нужна персоналу на входе — **не гостям**. У гостей фестиваля здесь **нет аккаунтов**, они сюда вообще не заходят. Их билеты лишь **сканируются** сотрудниками.

Кто работает в Baza:

| Кто | Что делает |
|-----|-----------|
| **Билетёр на воротах** | Сканирует QR-коды, впускает гостей |
| **Охрана** | То же — контроль доступа |
| **Инфоцентр** | Поиск гостя вручную (если QR не читается) |
| **Администратор** | Всё то же + ведёт смены, смотрит отчёты, делает синхронизацию данных |

Учётные записи сотрудников заводятся **заранее, вручную** (через сидер при настройке или прямо в базе данных). «Регистрации с улицы» нет.

## 3. Как билет попадает в Baza (кратко)

Сами билеты в Baza **не создаются**. Они приезжают из основного приложения **org (Backend)** в момент, когда заказ оплачен или билет выдан.

Ключевая особенность: org пишет билеты **напрямую в базу данных Baza** (через отдельное подключение к чужой БД), **а не через какой-то API**. Делается это асинхронно и с контролем доставки — если Baza временно недоступна, билет всё равно выдаётся гостю, а запись «доедет» повторными попытками.

```
   org (Backend)                         Baza (система входа)
   ┌──────────────┐   прямая запись в    ┌──────────────────┐
   │ заказ оплачен │ ───── чужую БД ────► │ el_tickets       │
   │ билет выдан   │   (соединение        │ spisok_tickets   │
   │              │    mysqlBaza)        │ live_tickets     │
   └──────────────┘                       │ auto             │
        │ контроль доставки               └──────────────────┘
        │ (модуль BazaDelivery,                     ▲
        │  до 10 повторных попыток)                 │ персонал сканирует
        ▼                                           │ QR на КПП
   ┌──────────────┐                                 │
   │ admin видит   │                          ┌─────┴──────┐
   │ застрявшие    │                          │ ScanController│
   │ доставки      │                          └────────────┘
   └──────────────┘
```

Подробнее об этом контракте — в разделе «Системная картина» проекта (модуль `Backend/src/BazaDelivery/`). Здесь важно лишь одно: **Baza получает уже готовые билеты и только читает их при сканировании**.

## 4. Сканирование и впуск гостя на КПП (пошагово)

Главный рабочий экран — **сканер** (страница `/scan`). На нём сразу включается камера (JS-библиотека QrScanner).

Пошаговый процесс впуска:

1. **Сотрудник открывает сканер.** Камера непрерывно читает QR-коды.
2. **QR распознан.** Браузер отправляет на сервер содержимое QR — это ссылка вида `https://org.spaceofjoy.ru/ticket/live/...` или `.../newTickets/<uuid>`.
3. **Сервер определяет тип билета.** Сервис `DefineService` разбирает ссылку: по подстроке в URL понимает, какой это тип билета (электронный / живой / парковка), и извлекает идентификатор (UUID для электронного, номер для живого/парковки).
4. **Билет ищется в базе.** Для живого билета номер сначала **расшифровывается** (он зашифрован в QR), потом ищется. Электронный билет ищется сразу в двух таблицах — обычных и списочных билетов.
5. **Если билет не найден** — экран показывает красную ошибку «Не найден билет…».
6. **Если найден** — появляется карточка гостя: имя, email, телефон, тип билета, дата заказа, статус, комментарий, цвет браслета.
7. **Решение «пускать или нет».** Кнопка **«ПРОПУСТИТЬ»** показывается **только если** выполнены **оба** условия:
   - билет **ещё не использован** (по нему никто не входил);
   - билет **оплачен/одобрен** (статус `paid` / `paid_for_live` / `approve_list`).

   Иначе вместо кнопки — плашка «Был пропущен `<дата>`» (если уже входили) или «Билет находится в статусе `<статус>`» (если не оплачен/отменён).
8. **Нажатие «ПРОПУСТИТЬ».** Сервер делает три шага:
   - находит **текущую открытую смену** этого сотрудника (если смены нет — ошибка «Ваша смена не найдена, обратитесь к Кристине»);
   - **прибавляет +1** к счётчику прошедших билетов нужного типа в отчёте смены;
   - **помечает билет впущенным** — записывает в него `change_id` (номер смены) и `date_change` (текущее время).
9. **Готово.** Экран показывает «Билет N прошёл», очищает карточку и снова запускает сканер для следующего гостя.

**Запасной путь — ручной поиск** (страница `/search`). Если QR не читается, сотрудник вводит строку (номер билета, имя, email, телефон, номер авто), и система ищет сразу по всем типам билетов. Результаты выводятся таблицей с вкладками по типам, у каждой строки — своя кнопка «Пропустить».

> ✅ **Исправлено (PR #84, 2026-06-19):** раньше проверка «можно ли впустить» жила только на фронте, и сервер при впуске не перепроверял, что билет уже использован (повторный запрос накручивал отчёт). Теперь `skip()` на сервере бросает ошибку, если билет уже пропущен, а инкремент отчёта идёт только после успешного впуска.

## 5. Типы билетов на входе

На входе встречается шесть разновидностей билетов. Сами билеты создаются в org — Baza их только распознаёт и проверяет.

| Тип | Что это простыми словами | Цвет браслета | Как ищется |
|-----|--------------------------|---------------|-----------|
| **Электронный** (`el`) | Обычный билет, купленный заранее на сайте, привязан к человеку (имя, email, телефон) | 🟢 зелёный | По QR (UUID) и вручную |
| **Заказ-список** (`spisok`) | Групповой въезд: куратор завёл список гостей на проект/локацию. Вместо платёжных данных — проект и куратор | 🔵 синий | Через QR (UUID списка) и вручную |
| **Дружеский / Friendly** (`friendly`) | Билет, проданный продавцом (роль pusher) | 🟢 зелёный | Только вручную (через QR не распознаётся) |
| **Живой** (`live`) | Билет, купленный прямо на месте — это заранее заготовленный пул бумажных номеров. Номер зашифрован в QR | 🟢 зелёный | По QR (после расшифровки номера) и вручную |
| **Парковка** (`parking`) | Въезд автомобиля. Бывает трёх подвидов: гостевая, «для своих» (бесплатная), «вездеход» | 🟢 зелёный | По QR и вручную |
| **Авто** (`auto`) | Машины кураторов/проектов — служебный въезд. Это не совсем билет гостя | ⚪ белый | Только вручную (по гос-номеру) |

**Цвет браслета** — это подсказка сотруднику, какой браслет выдать. Обычные гости (электронный/живой/дружеский) получают **зелёный**, групповой въезд по спискам — **синий**, служебные машины — **белый**.

> Замечания (для разработчиков): три зелёных типа визуально различаются только в карточке (по полю «тип»), а не цветом браслета. У парковки **нет своего** цвета — она тоже использует зелёный.

## 6. Смены (changes) и отчёты — подробно

> Этот раздел — **основа для грядущего обновления**, поэтому описан детально: что есть сейчас и чего пока нет.

### Что такое «смена»

**Смена** — это рабочий промежуток времени, когда один или несколько сотрудников стоят на входе и пропускают гостей. Одна запись смены содержит:

- **состав** — список сотрудников, дежуривших в смену (хранится как массив их id);
- **время начала** (задаётся вручную администратором) и **время окончания** (ставится автоматически при закрытии);
- **счётчики** — сколько билетов каждого типа прошло за смену;
- привязку к фестивалю.

### Жизненный цикл смены

```
   [Админ создаёт смену]                      [Сотрудники сканируют]              [Админ закрывает]
   ───────────────────►  СМЕНА ОТКРЫТА  ──────────────────────────►  СМЕНА ОТКРЫТА  ──────────────►  СМЕНА ЗАКРЫТА
   состав + время начала   (end = пусто)        счётчики растут +1       (end всё ещё пусто)            (end = время now)
```

- **Создание** (`/change/edit` → `/change/save`): админ выбирает из списка сотрудников состав смены и указывает дату/время начала. Создаётся запись, все счётчики = 0, окончание (`end`) пустое.
- **«Открыта»** — это просто факт существования смены с пустым полем `end`. Отдельного действия «открыть» нет (в коде осталась папка `Open`, но реально работает только закрытие).
- **Сканирование** наполняет счётчики (см. раздел 4): каждый впуск делает +1 к нужному типу.
- **Закрытие** (`/change/close`): проставляется время окончания. Смена больше не «текущая».
- **Удаление** (`/change/remove`): жёсткое удаление записи (без возможности восстановления и без журнала).

### Что считает смена сейчас (счётчики)

| Счётчик | Что считает |
|---------|-------------|
| `count_el_tickets` | Электронные билеты |
| `count_live_tickets` | Живые билеты |
| `count_drug_tickets` | Дружеские (Friendly) — `drug` = «друг» |
| `count_spisok_tickets` | Билеты из заказов-списков |
| `count_auto_tickets` | Авто (машины) |
| `count_parking_tickets` | Парковка платная |
| `count_parking_free_tickets` | Парковка бесплатная («для своих») |
| `count_parking_cross-country_tickets` | Парковка «вездеход» |

### Отчёт по сменам

Администратор открывает **`/report`** — таблицу всех смен: состав (имена сотрудников склеиваются через запятую), счётчики по типам, парковки, время начала/окончания, кнопку «Закрыть смену» (только для открытых) и строку **«Всего»** внизу.

> ⚠️ **Особенность итога (для разработчиков):** строка «Всего» считается **только** как `живые + дружеские + списки + электронные`. Парковки и авто **в общий итог не входят** (хотя суммируются отдельными колонками). Это нужно зафиксировать перед финансовым/количественным учётом.

### Чего в смене НЕТ сейчас (важно для апдейта)

Подтверждено отсутствием в структуре данных:

- **Нет финансов вообще** — ни выручки, ни сумм, ни наличных/безнала, ни инкассации. Ни одного денежного поля. (При этом Baza цен билетов **не знает** — они хранятся в org.)
- **Нет материальной ответственности** — нет понятия «кто отвечает за кассу/выручку/бланки», нет «старшего смены», нет приёма/сдачи денег или инвентаря.
- **Нет комментариев начальника смены** — нет поля для заметок/замечаний на смене.
- **Нет ролей внутри смены** — состав хранится плоским списком id, без различения «кассир / сканер / старший».
- **Нет учёта бланков браслетов** — только агрегатные счётчики «+1», без привязки к конкретным бланкам.
- **Нет аудита** — кто и когда открыл/закрыл/изменил смену, не фиксируется (в org для этого есть журнал `domain_history`, в Baza аналога нет).
- **Нет фиксированной длительности** — время начала задаётся вручную, окончание ставится по факту закрытия; планового времени и перерывов нет.

### Технические нюансы (для разработчиков)

- **Текущая смена сотрудника** = первая открытая (с пустым `end`) по дате старта. Если у человека по ошибке две открытые смены — впуск упадёт в самую раннюю.
- Счётчик `count_auto_tickets` есть в базе и в отчёте, но **отсутствует в списке `$fillable`** модели — он наполняется только операцией increment. При добавлении новых полей через `updateOrCreate` их нужно явно вносить в `$fillable`, иначе они молча не сохранятся.
- Привязка к фестивалю (`festival_id`) **жёстко зашита константой** в коде создания смены и в запросе отчёта — выбора фестиваля в интерфейсе нет.
- Имя колонки `count_parking_cross-country_tickets` содержит дефис — потенциальный источник ошибок в SQL (требует обратных кавычек).

## 7. Синхронизация и офлайн-работа на ноутбуке-сканере

На фестивале часто **нет устойчивого интернета**, поэтому ноутбук-сканер на КПП работает **автономно (офлайн)**, держа свою копию Baza. Чтобы перенести данные туда и обратно, есть подсистема **синхронизации** — перенос через файл-архив вручную («sneakernet»: данные едут не по сети, а в ZIP-файле, который человек переносит на флешке/по локальной сети).

```
   БОЕВОЙ СЕРВЕР Baza                              НОУТБУК-СКАНЕР на КПП (офлайн)
   ┌──────────────────┐                            ┌──────────────────────────┐
   │  /sync            │   1. Выгрузить ZIP         │  /sync                    │
   │  «Выгрузить ZIP»  │ ──── (export) ────►        │  «Загрузить ZIP» (import) │
   └──────────────────┘                            └──────────────────────────┘
        │                  2. перенос файла              ▲       │
        │                  (флешка / лок. сеть)          │       │ 3. сотрудники
        └────────────────────────────────────────────────┘       ▼   сканируют офлайн
                                                              отметки входа копятся
   ◄──────────────── 4. забрать отметки обратно тем же способом ──────────────────
```

### Как устроена выгрузка (export)

- Перебираются **6 разрешённых таблиц**: `auto`, `changes`, `el_tickets`, `live_tickets`, `parking_tickets`, `spisok_tickets`.
- Каждая таблица **построчно** (порциями по 1000 строк, чтобы не перегружать память) стримится в файл формата **NDJSON** (`.jsonl` — одна JSON-запись на строку).
- Все файлы упаковываются в один **ZIP** и отдаются на скачивание.
- То же самое доступно из командной строки: `artisan baza:export` (с опцией `--since` для выгрузки только изменённых строк).

### Как устроена загрузка (import)

- ZIP распаковывается, каждая строка читается отдельно.
- **Логика слияния (upsert) по id:** нет такой записи → вставить; есть и в файле она **новее** (по метке времени `updated_at`) → обновить; иначе → пропустить.
- **Удалённые на источнике записи не трогаются** (по требованию).
- Каждая строка обрабатывается в своём «защитном блоке»: битая строка или ошибка пропускается с записью в лог и **не валит весь импорт**.
- Лимит файла — **до 500 МБ**, только ZIP.

### Что синхронизация увозит обратно

Главная ценность обратной синхронизации — **отметки входа**, сделанные офлайн: `change_id` (к какой смене относится проход) и `date_change` (когда впустили). Именно они и есть «дельта», ради которой данные потом выгружают с ноутбука обратно на сервер.

### Реальность офлайна сейчас (важно)

- Код выгрузки/загрузки **есть и работает** симметрично.
- Но **автономный офлайн-клиент** (локальный SSL-сертификат для камеры-сканера, готовый офлайн docker-набор) — **ещё только в планах** (версия v2.8.0). Кода под это пока нет.
- На прошлом фестивале сканер **не заработал** именно из-за отсутствия HTTPS в локальной сети без интернета (камера в браузере требует защищённого соединения).

> ⚠️ **Открытые вопросы (для разработчиков):** при нескольких офлайн-ноутбуках слияние идёт по «последняя запись побеждает» (без объединения полей) — нужно подтвердить, что отметки входа не теряются. Также: отменённый на сервере билет на офлайн-ноутбуке останется валидным (синхронизация не удаляет записи) — приемлемо ли это для контроля входа.

## 8. Роли и права (текущие)

В Baza **очень простая, плоская модель прав** — всего **два уровня**, определяемых единственным булевым флагом `is_admin`:

| Роль | Флаг | Что доступно |
|------|------|--------------|
| **Обычный сотрудник** (билетёр, охрана, инфоцентр) | `is_admin = false` | Вход в систему, сканирование (`/scan`), ручной поиск и впуск (`/search`), редактирование своего профиля/пароля |
| **Администратор** | `is_admin = true` | Всё то же + отчёты по сменам (`/report`), создание/закрытие/удаление смен (`/change/*`), синхронизация данных (`/sync/*`) |

Особенности модели прав:

- **Других ролей нет.** Нет именованных ролей (как `seller`/`manager`/`curator` в org) — только «админ / не админ».
- Различия вроде «охрана» / «инфоцентр» / «билетёр КПП» существуют **только в имени** сотрудника, а на права **не влияют**.
- **Роли продавцов/кураторов** (`seller`, `curator`) в Baza — это **не учётки, а текстовые метки** в самих билетах: «кто создал этот билет в org».
- **Гостей в Baza нет** как класса пользователей.
- Вход устроен через **сессии Laravel** (классический логин по email/паролю), **не через JWT** (в отличие от org).

> ⚠️ **Известные технические нюансы (для разработчиков):**
> - ✅ **Исправлено (PR #84):** эндпоинты впуска (`/api/scan`, `/api/enter`) раньше были без auth-middleware, а id сотрудника приходил из тела запроса. Теперь они перенесены в `web`-группу под `middleware('auth')` (сессия + CSRF), а id сотрудника берётся из `Auth::id()`.
> - Страницы `/profile`, `/password-request`, `/users` **не закрыты** middleware `auth` на уровне маршрута (в отличие от `/scan` и `/search`). Профиль/пароль защищены проверкой внутри формы, а `/users` — потенциально доступна без логина.
> - Самостоятельная **регистрация персонала сломана**: форма есть, но обработчика для неё нет. Учётки заводятся только заранее.
> - В тестовом наборе у всех учёток пароль `secret` — это явно тестовые данные, на проде должны быть другие.

## 9. Схема БД (таблицы простыми словами)

База данных Baza — это упрощённая копия билетов из org плюс журнал смен и учётки персонала.

### Таблицы билетов

| Таблица | Что хранит | Ключевые поля |
|---------|-----------|---------------|
| **`el_tickets`** | Электронные (обычные) билеты — главная таблица входа | `kilter` (номер, по нему сканируют), `uuid` (= id билета из org, точная связь с заказом), имя/email/телефон/город, `status`, `change_id`, `date_change`, `number` (номер живого билета, обычно пусто), `festival_id` |
| **`spisok_tickets`** | Списочные билеты (групповые от кураторов) | `kilter`, `project` (проект), `curator` (куратор), email/имя получателя, `ticket_uuid` (ссылка на UUID из org), `status`, `festival_id` |
| **`friendly_tickets`** | Дружеские билеты (от продавца) | `kilter`, `project`, `seller` (продавец), email/имя гостя, `status`, `festival_id` |
| **`live_tickets`** | Живые билеты (выдаются на месте) | `kilter` (= бумажный номер, шифруется в QR), `el_ticket_id` (ссылка на электронный билет, если связан), `status`, `change_id`, `date_change` |
| **`parking_tickets`** | Парковочные билеты | `kilter`, `type` (подтип парковки), `status`, `change_id`, `date_change` |
| **`auto`** | Авто-пропуска (машины кураторов) | `order_id` (UUID заказа-списка из org), `curator`, `project`, `auto` (гос-номер), `festival_id` |

**Поля-метки входа** (есть у всех билетов): `change_id` — к какой смене отнесён проход; `date_change` — когда впустили. Если они **пусты** — билет ещё не использован.

**Идентификация между org и Baza** идёт по **UUID**, а не по числовому id: `el_tickets.uuid`, `spisok_tickets.ticket_uuid`, `live_tickets.el_ticket_id`, `auto.order_id` — все ссылаются на сущности из org. Это позволяет точно найти запись при отмене (поставить `status = cancel`).

### Служебные и журнальные таблицы

| Таблица | Что хранит |
|---------|-----------|
| **`changes`** | Смены дежурства: состав (`user_id` — JSON-массив id сотрудников), `start`/`end`, `festival_id`, счётчики прошедших билетов по типам |
| **`users`** | Сотрудники Baza: id, имя, email (логин), пароль, `is_admin` (флаг администратора) |
| **`password_resets`** | Токены сброса пароля |
| **`personal_access_tokens`** | API-токены (Laravel Sanctum) — подключены к модели, но боевого входа по токену нет |
| **`failed_jobs`** | Упавшие фоновые задачи (служебный журнал) |
| **`migrations`** | Список применённых миграций (стандартная служебная таблица Laravel) |

> ⚠️ **Нюансы схемы (для разработчиков):**
> - `festival_id` хранится по-разному в разных таблицах (где-то `char(36)` nullable, где-то NOT NULL, где-то `varchar(255)`), а в `live_tickets` и `parking_tickets` его **нет вообще** — эти билеты не разделяются по фестивалям на уровне поиска (риск показать прошлогодний билет).
> - **Нет внешних ключей (FK)** между таблицами на уровне БД — целостность держится только на уровне приложения.
> - Дамп `baza.sql` в репозитории содержит **реальные персональные данные** гостей (ФИО, email, телефоны) и хеши паролей — вопрос 152-ФЗ и хранения в git.

## 10. Интерфейс (текущие экраны)

Интерфейс Baza сделан на **серверных страницах Laravel (Blade)** с готовой Bootstrap-темой «Black Dashboard» и jQuery. Это **не** современный Vue/SPA (в отличие от новой админки org).

| Экран | Маршрут | Кому | Что делает |
|-------|---------|------|-----------|
| **Логин** | `/login` | всем | Вход персонала по email + пароль («Добро пожаловать на смену») |
| **Сканер QR** | `/`, `/scan` | сотрудник | Камера + карточка гостя + кнопка «ПРОПУСТИТЬ». Главный рабочий экран |
| **Поиск** | `/search` | сотрудник | Ручной поиск (когда QR не читается), результаты по вкладкам: Электронные / Списки / Френдли / Живые / Авто |
| **Смены** | `/report` | админ | Таблица смен: состав, счётчики по типам, время, «Закрыть смену», итоги |
| **Добавить/изменить смену** | `/change/edit` | админ | Мультивыбор состава смены + время начала, сохранить/удалить |
| **Синхронизация** | `/sync` | админ | Две карточки: «Выгрузить ZIP» и «Загрузить ZIP» + статистика импорта |
| **Профиль** | `/profile` | всем | Редактирование имени/email + смена пароля |

**Навигация:** левый сайдбар у всех показывает «Поиск» и «Сканер QR»; пункты «Смены» и «Синхронизация» видны только администратору.

> ⚠️ **Технические замечания (для разработчиков):**
> - В шаблонах **захардкожены ссылки** на прод-домен `https://vhod.spaceofjoy.ru/` вместо относительных путей — это ломает работу на стенде и на офлайн-ноутбуке.
> - Есть **мёртвые страницы** из коробки темы (демо-дашборд с заглушками «Dakota Rice», `$36 738` и т.п.) — в боевом меню не используются, кандидаты на удаление.
> - Подпись роли «Administrator» в профиле захардкожена и показывается **всем** независимо от реальной роли.
> - Маршрут `/users` по ошибке возвращает экран смен, а не список пользователей — похоже на незавершённую заглушку.

## 11. Что меняется в грядущем обновлении

> ⚠️ **Всё, что ниже — ПЛАНЫ, требующие уточнения.** Сейчас этих возможностей в коде НЕТ (подтверждено отсутствием полей в структуре данных). Для большинства пунктов потребуются новые таблицы/поля и техническое задание.

Планируемые направления:

- **Новый интерфейс (UI).** Возможный переход с серверного Blade + jQuery на современный стек (как новая админка org на Vite + PrimeVue) — либо доработка текущего. *Требует решения.*
- **Полноценная система смен.** Сейчас смена — это лишь таблица счётчиков + состав. Планируется развить: графики, отметки прихода, расписания. *С нуля.*
- **«Финансы и отчёты за смену».** Сейчас финансов нет вообще. Нужно ТЗ: какие именно деньги считать на смене и **откуда брать суммы** (Baza цен билетов не знает — они в org/qr). *Требует источника данных и ТЗ.*
- **Материальная ответственность.** Учёт того, за что отвечает смена: **бланки браслетов** и **деньги на обмен/размен на КПП**, приём и сдача матценностей. Нужна новая сущность «ответственный» и, вероятно, аудит/журнал. *С нуля.*
- **Привязка браслета к билету/заказу через QR.** На самом браслете печатается **отдельный QR-код**, по которому в любой момент можно **отследить владельца билета** — кто это, какой у него билет и заказ. Браслет становится «физическим продолжением» билета: отсканировал браслет на территории → увидел, кому он принадлежит. *С нуля (нужна новая связь браслет ↔ билет/заказ).*
- **Новые роли** (вместо текущей плоской «админ / не админ») — всего **пять**:
  - **Администратор** — ввод данных о сменах, просмотр отчётов, создание пользователей, одобрение «зависших» билетов. *Функционал в целом уже есть — нужно привести к единому стандарту.*
  - **Начальник смены** — главный по смене, несёт за неё ответственность: видит и **корректирует отчёт за смену** (с комментариями), **принимает материальные ценности** от прошлой смены и от штаба, ведёт **финансовую отчётность**.
  - **Билетёр** — рядовой работник смены: базовый функционал — сканирование QR, поиск и связывание билетов.
  - **Комендант КПП** — член команды на штабе, который **принимает смены**, может их менять и менять их состав; передаёт на КПП материальные ценности (браслеты, бланки, деньги на размен). Плановая длительность смены — **10 часов**. Визуализация — сетка «точки КПП × смены» (как школьное расписание).
  - **Охранник** — для него **основная функция** — отслеживание владельца билета по **QR-коду на браслете** (кто это, какой билет/заказ). Та же функция доступна **и всем остальным ролям**, но у охранника она ключевая.

  Для ролей внутри смены текущая структура (плоский список id) **не подойдёт** — понадобится отдельная связующая таблица «смена ↔ сотрудник ↔ роль».

**Сопутствующие технические задачи при апдейте** (вытекают из текущих ограничений):
- ввести **аудит** действий со сменами (как `domain_history` в org);
- заменить жёсткое удаление смены на **мягкое** (soft delete), чтобы не терять финансовые данные;
- вынести `festival_id` из жёсткой константы в **web-интерфейс** выбора фестиваля (+ добавить фильтр в live/parking, где его сейчас нет);
- отчёт разделить на **два отсека** — браслеты (люди) и машины (парковки), упразднив текущий неполный «Всего»;
- развязать домены: запись билетов **через ingest-API Baza** (fallback — прямая БД), вебхук «билет прошёл» обратно в org для истории;
- ✅ закрыть **security**-дыру впуска (auth + серверная проверка «уже впущен», `user_id` из сессии) — **сделано (PR #84)**, см. §4/§8.

Там же — финансы смены, материальная ответственность (браслеты/бланки/инкассация/довоз/залоговая парковка), привязка браслет↔билет через QR, чат смены через WebSocket, мобильный офлайн-интерфейс.
