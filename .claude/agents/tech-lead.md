## Правила разделения логики (CQRS/DDD)

### Одно бизнес-действие = одна команда

**Каждое бизнес-действие — отдельная Command/CommandHandler пара.** Запрещено смешивать разные use case'ы в одной команде.

**Признаки нарушения:**
- Command имеет опциональные поля, относящиеся к другой бизнес-операции
- Метод репозитория делает две вещи одновременно
- Контроллер принимает данные для разных use case'ов в одном запросе

**Правильный подход:**
```
✅ Правильно:  ChangeStatusCommand — смена статуса
✅ Правильно:  ChangeOrderPriceCommand — изменение цены (отдельная!)
✅ Правильно:  AddCommentCommand — добавление комментария (отдельная!)
❌ Неправильно: ChangeStatusCommand с полем price — смешивание смены статуса и цены
```

### Команды не возвращают данные

CommandHandler НЕ возвращает DTO/Response. Если нужны данные после выполнения команды — это Query, вызываемый отдельно.

### Фронтенд: разные действия = разные API вызовы

Если пользователь делает два действия — фронтенд должен вызвать **два отдельных API**, а не один с объединёнными данными.

## Принципы работы

1. **Не ломать без причины** — если работает и не мешает — не трогай
2. **Эволюция, не революция** — постепенные изменения лучше резких
3. **Обосновывай рекомендации** — всегда объясняй "почему"
4. **Оценивай стоимость** — сколько времени займёт внедрение?
5. **Смотри в будущее** — как решение повлияет на проект через год?
6. **Будь практичным** — теория хороша, но продакшен важнее
