# Ф5 — Offline-first PWA BazaFront (план реализации)

> **Статус:** дизайн-гейт СНЯТ владельцем (2026-06-20). Готово к реализации сериями PR.
> **Контекст:** критический путь к надёжному офлайн-входу на КПП (план `baza-update-implementation-plan.md` §Ф5). Greenfield рядом со старым Blade (Strangler) — старый `/scan`,`/search`,`/enter` остаётся fallback ≥1 фестиваль, cutover после паритета и доказанного офлайна.

## Решения владельца (дизайн-гейт, 2026-06-20)

| Вопрос | Решение |
|--------|---------|
| **Главный экран** | **Сразу сканер (камера)**. Шапка: текущий КПП/смена. Снизу таб-бар: Скан / Поиск / Смена. Под камерой — «ввести № вручную». Максимум скорости на входе. |
| **MVP-состав Ф5** | **Только горячие экраны: сканер + впуск + поиск** (без QR). На готовых API `/api/scan`, `/api/enter`, `/search` (PR #84). Офлайн-first. |
| **Сетка «КПП × смены»** | Задумана **интерактивной** (назначение людей по КПП/сменам), но **НЕ в MVP Ф5** — отдельным срезом позже (пересекается с формой состава смены Ф2). |
| **Быстрая живая продажа** | **Отложена в Ф7** (финансы/матответственность, гейт Q4). В Ф5 продажи нет. |

## Целевое устройство
Телефон (портрет), крупные кнопки, мгновенный отклик, тач. Камера требует secure context (HTTPS) → локальный SSL (mkcert/локальный CA).

## Архитектура (из плана §Ф5, уже зафиксирована)
- **Отдельное приложение** `BazaFront/` (Vite + PrimeVue), **своя session-auth** (не JWT, не AdminFront) — логинится в Baza, использует её сессию/куки.
- **Docker-сервис** `node-baza`, раздача под `/baza/` (как `AdminFront` под `/admin/`).
- **Offline-first:** service worker (кэш оболочки + install на телефон) + IndexedDB-очередь операций (отметки впуска копятся офлайн, дренаж при сети).
- **CI-job сборки** ЗАРАНЕЕ (чтобы не повторить TD-32 — CI не собирал AdminFront).
- Старый Blade (`/scan`,`/search`,`/enter`) — fallback, не трогаем до cutover.

## Разбивка на PR (серия)

1. **PR-1 — Каркас + инфра (ungated):** `BazaFront/` (Vite + PrimeVue, роутер, session-auth через куки Baza), Docker-сервис `node-baza`, nginx-раздача `/baza/`, **CI-job сборки**, шаг сборки в `deploy-staging.yml`. Пустой каркас + 1 экран-заглушка против живого Baza-API. Критерий: открывается на `vhod.staging/baza/`, логин по сессии Baza.
2. **PR-2 — Service worker + офлайн-оболочка:** кэш статики/оболочки, install-на-телефон (manifest), IndexedDB-схема очереди. Критерий: оболочка открывается при отрезанном интернете.
3. **PR-3 — Экран «Сканер»:** камера (QrScanner, локальный — уже есть в Baza Ф0) → скан → распознать билет (`/api/scan`) → карточка → «Пропустить» (`/api/enter`). Офлайн: отметка впуска в IndexedDB-очередь + дренаж при сети. Главный экран.
4. **PR-4 — Экран «Поиск без QR»:** ручной поиск (`/search` → `ticket_search`, готов в rich-search) → карточка → «Пропустить». Для гостей без QR.
5. **PR-5 — Локальный SSL + cutover-подготовка:** mkcert/локальный CA + runbook установки корневого серта на телефоны; проверка камеры офлайн на локальном SSL. Параллельная жизнь со старым Blade.
6. **(позже)** сетка «КПП × смены» (интерактивная), быстрая продажа (Ф7) — вне MVP Ф5.

## Открытые вопросы к реализации (по ходу)
- Session-auth: как BazaFront под `/baza/` шарит сессию со старым Blade (один домен `vhod.*`, общие куки) — проверить CSRF/SameSite.
- Дренаж офлайн-очереди впуска: тот же механизм, что outbox Ф4 (`baza_entry_outbox`) или клиентский IndexedDB → POST `/api/enter` при сети. Скорее клиентский (отметки делаются на телефоне офлайн).
- Бренд/тема — со staging (фирменный стиль Solar Systo, как договорено для AdminFront).
