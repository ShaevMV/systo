## Целевое устройство
Телефон (портрет), крупные кнопки, мгновенный отклик, тач. Камера требует secure context (HTTPS) → локальный SSL (mkcert/локальный CA).

### Рекомендуемая архитектура (итог)
- **Переезжает ТОЛЬКО UI (Blade→PWA).** Laravel-Baza остаётся **API-ядром** (уже Laravel 11 / Sanctum 4, чистая CQRS, БД только в репозиториях — UI это деталь по Dependency Rule). Бэкенд НЕ переписываем, только **аддитивно расширяем**. ✅ Подтверждено владельцем 2026-06-20.

#### ⚠️ КОРРЕКТИРОВКА владельца 2026-06-20 (отвечает на отчёт PM) — топология без гарантированного ноута

Владелец уточнил реальность феста — это меняет 3 допущения PM:

- **Ноутбука на КПП может НЕ БЫТЬ.** → Нельзя делать локальный узел КПП **единственным** источником истины. Узел-ноут — **опциональный ускоритель**, а не обязательное звено.
- **Синк физическими носителями невозможен** для телефонов («в телефон флешку не вставишь»). → Sneakernet-ZIP остаётся каналом только **между узлами/ПК** (штаб↔сервер), но НЕ способом синхронизации телефонов. Телефоны синкаются **только беспроводно**.
- **Локальный Wi-Fi есть, но работает не всегда** (особенно «в военное время» — глушилки/обрывы). → Синк должен пережить **полное отсутствие связи**, а при любом окне связи — досылать оппортунистически.

**Итоговая модель (laptop-optional, cloud-eventual):**
1. **Каждый телефон автономен офлайн** — держит: (а) кэш оболочки (service worker), (б) **минимизированный шифрованный снимок билетов** в IndexedDB (только поля впуска, B5), (в) **append-only журнал намерений впуска** (PR-8). Может впускать гостей **только по локальным данным**, без какой-либо связи.
2. **Источник eventual-истины = боевой сервер (облако), синк по ЛЮБОМУ интернету** — сотовые данные телефона, а не только локальный Wi-Fi. Как только у телефона появляется *любой* интернет — он пушит свой журнал и тянет чужие отметки. Облако off-site → не зависит от падения локального Wi-Fi.
3. **Опциональный локальный узел (ноут/ПК), если есть** — ускоряет синк по локальному Wi-Fi (без сотовой), кэширует снимок. Но система **полностью работает и без него** (телефоны ↔ облако напрямую).
4. **Браслет = механизм консистентности на ПОЛНЫЙ блэкаут.** Когда связи нет вообще (ни сотовой, ни Wi-Fi, ни узла) — два КПП физически не видят впуски друг друга в реальном времени. Дыру закрывает **физический браслет**: получив браслет на любом КПП, гость визуально уже впущен; цифровые журналы сверяются позже при возврате связи («первый впуск побеждает», PR-8). Это и есть зачем браслеты (Ф6) — не только материальный учёт, но и **офлайн-anti-double-entry на время блэкаута**.
- **Auth: онлайн-вход = session-куки Baza + CSRF** (НЕ JWT); **офлайн-вход = PIN** (Argon2id, разблокирует AES-GCM ключ кэша); привязка к устройству — **опционально** поверх (подтверждено владельцем). ⚠️ Прежняя формулировка «session-auth через куки офлайн» — техошибка (кука валидируется сервером, офлайн бесполезна).
- **Отметки впуска — append-only** (НЕ last-write-wins по строке билета; `ImportApplication::isFresher` = LWW по `updated_at` остаётся только для каталога билетов, который мастерит org).
- **Кросс-браузер обязателен.** Телефоны личные, браузеры разные → не Chrome-only. Опираемся на стандартный PWA-стек (service worker + IndexedDB + `getUserMedia` — поддержаны во всех современных мобильных браузерах). Сканер требует HTTPS (secure context); фолбэк при отказе камеры — ручной ввод № (экран «Поиск», PR-5).
- **Стратегия — Strangler (единогласно 4/4), big-bang отвергнут.** Старый Blade — fallback ≥1 фестиваль; cutover только после паритета + доказанного офлайна на реальном железе. Подробнее о Strangler для Baza — см. §«Strangler-план» ниже.

## Архитектура приложения
- **Отдельное приложение** `BazaFront/` (Vite + Vue 3 + PrimeVue, по образцу `AdminFront/`, но БЕЗ JWT-обвязки — `axios.withCredentials=true` + `X-XSRF-TOKEN`, session-куки Baza).
- **Docker-сервис** `node-baza`, раздача под `/baza/` (как `AdminFront` под `/admin/`).
- **Offline-first:** service worker (Workbox, `registerType:'prompt'` — НЕ autoUpdate) + IndexedDB (минимизированный шифрованный снимок + очередь намерений впуска).
- **CI-job сборки** ЗАРАНЕЕ (закрывает TD-32 первым PR).
- Старый Blade (`/scan`,`/search`,`/enter`) — fallback, не трогаем до cutover.

## Strangler-план (детально, по запросу владельца 2026-06-20)

**Strangler Fig** — замена легаси по кускам, без «большого взрыва» и без простоя: новая система растёт рядом со старой и постепенно «душит» (перехватывает) её функции, пока не заменит полностью. Старая всё это время — рабочий fallback.

Конкретно для Baza (UI Blade → PWA):

1. **Сейчас:** старый Blade-UI Baza (`/scan`, `/enter`, `/search`, server-rendered Laravel) — боевой. Новый PWA `BazaFront` поднят **рядом** на `/baza/` (✅ PR-1 на стенде). Оба ходят в **один и тот же Laravel-API** (`/api/scan`, `/api/enter`, `/search`) — API не дублируется.
2. **Душим по одному экрану:** доводим в PWA сначала **сканер** (PR-4), потом **поиск** (PR-5). Каждый — на готовом API. Пока экран не доказан — Blade-версия остаётся основной.
3. **Переключение маршрута (cutover, PR-10):** когда PWA-экран достиг паритета и **отработал офлайн на реальном фестивале**, перекидываем nginx-маршрут (`/scan`→PWA) — по одному экрану, не всё сразу. Старый Blade остаётся доступным как **аварийный fallback ещё 1–2 фестиваля** (просто не маршрутизируется по умолчанию).
4. **Удаление:** Blade-страницы убираем только после того, как PWA отработал фестиваль(и) без отката. **Laravel-API-ядро НЕ трогаем** — «душится» только UI-слой.

Почему так (а не переписать целиком): на КПП нельзя позволить себе нерабочий вход. Всегда есть откат на проверенный Blade; миграция обратима на каждом шаге; нет фест-окна с риском «всё легло».
