# Спецификация: пользователи, роли и права на каждый элемент админки (RBAC)

> **Статус:** черновик к согласованию · **Дата:** 2026-06-18 · **Автор:** Claude
> **Контекст:** при переезде админки на новые рельсы (AdminFront, Vite+Sakai) владелец попросил: «нужно всё же сделать возможность создавать пользователей и давать им роли. Тут я бы хотел иметь систему правил на каждый элемент». Решено: **сначала спроектировать права отдельно**, потом строить (перенос остальных экранов идёт параллельно).
> **Связано:** `.claude/specs/admin-frontend-vite-sakai.md` (переезд админки), `project_admin_ui_primevue`.

---

## 1. Что есть сейчас (факт по коду)

| | Состояние |
|---|---|
| **Роли** (`Backend/src/User/Account/Helpers/AccountRoleHelper.php`) | `guest`, `admin`, `seller`, `pusher`, `manager`, `curator`, `pusher_curator`, `qr_service` (сервис-аккаунт qr). |
| **Проверка доступа** | Только грубая: middleware `IsAdmin` (`is_admin` или `role=admin`), `CheckRole` (`role:seller,admin,…`), `IsSeller`. **Нет** модели прав на элементы. |
| **Аккаунты (admin API)** `Backend/routes/account.php` | `getList`, `getItem/{email}`, `edit/{id}`, `changeRole/{id}` — все admin. **Создания пользователя admin-эндпоинтом НЕТ.** |
| **Создание юзера** | Только публичные `POST /api/register` (роль `pusher`) и `POST /api/registerCurator` (роль `curator`). `is_admin`/`role` из тела игнорируются (защита). |
| **Поля User** | `name`, `email`, `role`, `city`, `phone`, `password`. |
| **Фронт (AdminFront)** | Гейтинг по роли через `meta.role: ['admin']` в роутере + проверка `appUser/isCorrectRole` (`POST /api/isCorrectRole`). Меню статично (не зависит от прав). |

**Вывод:** «система правил на каждый элемент» — это **новая фича**, а не перенос. В коде нет ни таблицы прав, ни энфорсмента по элементам.

## 2. Модель прав (предложение)

### 2.1 Элементы (что защищаем)
Элемент = раздел/экран админки. Каталог (ключи):

`dashboard`, `qr_orders`, `orders`, `orders_friendly`, `orders_lists`, `questionnaires`, `ticket_types`, `options`, `questionnaire_types`, `types_of_payment`, `locations`, `promo_codes`, `templates`, `template_bindings`, `email_delivery`, `users`.

### 2.2 Действия
`view`, `create`, `edit`, `delete` + спец-действия там, где они есть:
- `orders*` → `change_status` (смена статуса заказа).
- `questionnaires` → `approve` (одобрение анкеты).
- `users` → `assign_role` (назначение роли — самое чувствительное).

### 2.3 Связка
**Право = (роль, элемент, действие).** Матрица «роль × элемент × действие». `admin` — суперправо (всё разрешено, не редактируется). Остальные роли получают набор галочек.

> Вариант с правами **на пользователя** (override поверх роли) — НЕ предлагаю в v1: усложняет, а ролей достаточно. Можно добавить позже.

### 2.4 Хранение (БД, под CQRS/DDD проекта)
Таблица `role_permissions`:
```
role         varchar   -- из AccountRoleHelper
element      varchar   -- ключ из §2.1
action       varchar   -- из §2.2
PRIMARY KEY (role, element, action)
```
Наличие строки = право есть. Отсутствие = запрещено. `admin` в таблицу не пишем — он всегда «всё можно» (проверка короткозамкнута в коде). Модуль `Backend/src/Permission/` (пассивная сущность, как `Location`): DTO + Repository + Application (`getMatrix`, `setMatrix(role, [...])`, `can(role, element, action)`). БД только в репозитории.

## 3. Энфорсмент

### 3.1 Бэкенд (источник правды — обязателен)
- Новый middleware `permission:<element>,<action>` (по образцу `CheckRole`): берёт `Auth::user()->role`, спрашивает `PermissionApplication::can()`. `admin` → пропуск. Иначе нет права → `403`.
- Навешиваем на admin-роуты вместо/вместе с `admin`-middleware там, где нужна гранулярность. На первом этапе можно оставить `admin` как есть и добавлять `permission:` точечно.
- **Важно:** фронтовый гейтинг — это UX, НЕ безопасность. Реальная защита — на бэке.

### 3.2 Фронт (AdminFront)
- При логине грузим права текущего юзера: новый `GET /api/v1/account/myPermissions` → `{ permissions: ["orders:view","orders:change_status",…] }` (или матрица). Кладём в Vuex `appUser`.
- Composable `usePermissions()` → `can('orders','edit')`. Директива `v-can="'orders:edit'"` для кнопок.
- Меню `AppMenu.vue` фильтруется: пункт виден если `can(element,'view')`.
- Роутер `beforeEach`: `meta.permission = 'orders:view'` → нет права → редирект/403-экран.

## 4. Пользователи и роли (экран «Доступ»)

Раздел меню **«Доступ»** (виден при `users:view`):

1. **Список пользователей** — `POST /api/v1/account/getList` (есть). Колонки: имя, email, телефон, город, роль (Tag), действия.
2. **Создание пользователя** — ⚠️ **нужен новый бэкенд-эндпоинт** `POST /api/v1/account/create` (admin): `{ name, email, phone, city, password, role }`. Сейчас admin-создания нет (только публичный `register` с ролью `pusher`). Альтернатива-костыль: `register` + сразу `changeRole` — но это 2 запроса и роль `pusher` мелькает; лучше честный `account/create`.
3. **Редактирование** — `POST /api/v1/account/edit/{id}` (есть): имя/телефон/город.
4. **Смена роли** — `POST /api/v1/account/changeRole/{id}` (есть): `{ role }` (защита `assign_role`).
5. **Матрица прав** (Phase 2) — таблица «роль × элемент × действие» с галочками → `PermissionApplication::setMatrix`. Редактируется только при наличии права (или только `admin`).

## 5. Фазирование

| Фаза | Объём | Бэкенд | Фронт |
|---|---|---|---|
| **Phase 1 — Пользователи + роли** (S–M) | Экран «Доступ»: список/создание/редактирование/смена роли. Гейтинг меню/роутов по роли (как сейчас `meta.role`, но из данных). | Новый `account/create` (admin). | `UsersListView` (CRUD + changeRole). |
| **Phase 2 — Гранулярные права** (M–L, новая фича) | Матрица «роль × элемент × действие» + энфорсмент. | Модуль `Permission/` + таблица `role_permissions` + middleware `permission:` + `myPermissions` + миграция. | `usePermissions` + `v-can` + фильтр меню/роутов + редактор матрицы. |

> Phase 1 даёт то, что владелец явно просил (создавать юзеров + давать роли). Phase 2 — собственно «правила на каждый элемент».

## 6. Открытые вопросы (нужны ответы перед стартом)

1. **Гранулярность действий.** Достаточно `view/create/edit/delete` + спец (`change_status`/`approve`/`assign_role`)? Или нужнее проще — только `view`+`manage` (смотреть/менять) на элемент?
2. **Кто реально работает в новой админке?** org становится admin-only. Логинятся ли туда `seller`/`manager`/`curator`, или фактически только `admin` + 1-2 роли? От этого зависит, нужна ли вообще гранулярность сейчас или хватит ролей.
3. **Права на пользователя** (override поверх роли) — нужны или хватит «по роли»?
4. **Создание юзера** — делаем честный `account/create` (рекомендую) или костыль `register`+`changeRole`?
5. **Самозащита.** Запрещаем ли менять роль самому себе / снимать последнего админа (защита от «закрыл себе доступ»)?
6. **Дефолтная матрица.** Какие права у не-admin ролей по умолчанию (пустые → ничего, кроме явно выданного)?

---

## 7. Рекомендация

- Сделать **Phase 1 сразу после согласования** (это прямой запрос владельца, малый объём, нужен только `account/create` на бэке).
- **Phase 2** (матрица прав) — отдельным этапом, как полноценную фичу с бэкенд-модулем `Permission/` под CQRS. Не тащить её в текущий перенос экранов.
- Гранулярность — по **роли × элемент × действие** (не по пользователю), элемент = экран. Это покрывает «правила на каждый элемент» без переусложнения.
