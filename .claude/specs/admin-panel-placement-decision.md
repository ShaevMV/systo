# Решение: размещение админ-панели Baza и разграничение доменов org ↔ Baza

> **Статус:** решение архитектора (свод 4 независимых перспектив + факты кода на 2026-06-19).
> **Контекст-документы:** [BAZA.md](../docs/BAZA.md), [SYSTEM_OVERVIEW.md](../docs/SYSTEM_OVERVIEW.md), [baza-shift-finance-update.md](baza-shift-finance-update.md).
> **Твёрдые ограничения владельца:** **D14** — доменные модели org и Baza остаются несвязанными (максимум — передача билетов + вебхук «билет прошёл»); **D15** — работа в основном с мобильных, часто без интернета, на слабых устройствах.

## 0. Решения владельца (2026-06-19) — главенствуют

> **Главный принцип:** **Baza (система входа) работает НЕзависимо от org.** Даже если org полностью лежит — вход на фестиваль обязан продолжать работать. Это перевешивает аргумент о выносе аудита/хранения в org.

## 1. Контекст

Baza — система контроля входа (КПП) на фестиваль: персонал сканирует QR билета камерой, решает «пускать/нет», отмечает проход и ведёт смены. Сегодня это отдельное Laravel-приложение (Blade + jQuery + Bootstrap 4, тема Black Dashboard) на `vhod.spaceofjoy.ru`, session-auth, плоская роль `is_admin`. Билеты приезжают из org **прямой записью в чужую БД** (соединение `mysqlBaza`, неявный контракт через `TicketResponse::toArrayForBaza()/toArrayForSpisok()`); ingest-API у Baza нет, вебхука обратно в org нет.

Готовится крупное обновление Baza: 5 ролей вместо `is_admin`, финансы смены (живые продажи нал+безнал), материальная ответственность (браслеты/бланки/инкассация/залог), привязка браслет↔билет через QR, развязка доменов. Возник вопрос: **где разместить управление сменами/входом — оставить в Baza, перенести в новую org-админку AdminFront, или гибрид?**

Ключевые факты из кода (на 2026-06-19):
- Оперативный контур (сканер/впуск/смены/синк) физически работает на КПП, **в т.ч. офлайн на ноутбуке** (sneakernet ZIP-синк, 6 таблиц, upsert по `updated_at`).
- На прошлом фестивале сканер **не заработал** из-за отсутствия HTTPS в локальной сети без интернета (камера требует secure context).
- В Baza **нет финансов, нет матответственности, нет аудита смен, нет soft-delete** — большая часть обновления с нуля.
- org AdminFront работает на **JWT против сетевого Backend** → на офлайн-КПП недоступен в принципе.
- Безопасность впуска уже починена (PR #84): `/api/scan` + `/api/enter` под `auth`, `user_id` из `Auth::id()`, серверная защита от повторного впуска.

## 2. Рассмотренные варианты

### Вариант 1 — оставить управление сменами/входом в Baza (как сейчас, но развить)
### Вариант 2 — перенести/продублировать в org AdminFront (Vite + PrimeVue, JWT)
### Вариант 3 — гибрид: оперативный контур в Baza (офлайн-first), сводная аналитика read-only в org AdminFront

| Критерий | Вариант 1 (всё в Baza) | Вариант 2 (всё в AdminFront/org) | Вариант 3 (гибрид) |
|---|---|---|---|
| **Офлайн-работа КПП (D15)** | ✅ работает | ❌ JWT-SPA против сетевого Backend — мёртвый экран без сети | ✅ оперативный контур офлайн в Baza |
| **D14 (домены несвязаны)** | ✅ соблюдён | ❌ тянет домен входа/смены/финансов в org, связывает два auth | ✅ соблюдён (узкий контракт) |
| **Единый современный UI** | ❌ устаревший Blade+jQuery | ✅ единый AdminFront | 🟡 PWA Baza на общем визуальном языке + read-витрина в org |
| **Аудит/152-ФЗ хранение ПДн** | ❌ в Baza нет аудита, удаление жёсткое | ✅ domain_history в org | ✅ аудит/долгое хранение синкается в org |
| **Источник финансов живых продаж** | ✅ рождаются на КПП | ❌ org их не считает (Baza цен не знает) | ✅ пишет Baza, аудит в org |
| **Риск дня фестиваля** | 🟡 устаревший UI, нет аудита | 🔴 нет сети = нет входа (повтор провала) | 🟢 минимум точек отказа в горячем пути |
| **Объём работ** | 🟡 развитие в старом стеке | 🔴 переписать + связать два auth + синк смен в org | 🟡 PWA + контракт + финансы с нуля |

**Pros/cons по вариантам:**

- **Вариант 1** — *pro:* самодостаточность, офлайн, минимум связности; *con:* устаревший стек, плоская роль, нет аудита, URL захардкожены на прод-домен.
- **Вариант 2** — *pro:* единый UI/роли/дашборд; *con:* НЕ работает офлайн (фатально для КПП), нарушает D14, два домена auth в самом хрупком месте, дублирование sync-машины в org. **Отклонён единогласно всеми 4 перспективами.**
- **Вариант 3** — *pro:* офлайн-надёжность оперативного контура + защищённый долгоживущий аудит/аналитика; *con:* нужен второй фронт Baza (PWA), формализация контракта, финансы с нуля.

## 3. Рекомендация и обоснование

**Выбран Вариант 3 (гибрид) с владением оперативного контура в Baza.**

Оперативная панель КПП (сканер, впуск, смены, состав/роли смены через `change_user`, живые продажи, матответственность/браслеты/бланки/залог, привязка браслет↔билет) **остаётся и развивается внутри Baza** как offline-first инструмент. org AdminFront получает **только read-only витрину**: сводные дашборды продаж/проходов (AF-2), отчёты для бухгалтерии, алертинг застрявших доставок (C13).

**Почему именно так (обоснование с источниками):**

1. **Реальность эксплуатации — решающий аргумент (D15).** Смены создаются/закрываются, впуск и живые продажи происходят физически на КПП, часто без интернета. Код домена обязан исполняться там, где он работает. Это привязка кода к месту выполнения, не вкусовщина.
2. **Bounded Contexts + Dependency Rule (Р. Мартин, «Чистая архитектура», глава «Зависимости»).** «Смена/проход/браслет/касса смены» — единый Ubiquitous Language одного контекста → одна кодовая база-владелец. Домен входа НЕ должен зависеть от org (org для него — деталь-источник проекции билета, не ядро). Вариант 2 инвертировал бы зависимость в неправильную сторону.
3. **Решение владельца D14 уже задаёт ответ.** Несвязанные домены → домен смен/финансов КПП не переезжает в org; связь только через передачу билетов + вебхук «билет прошёл».
4. **Финансы рождаются на КПП.** org их источником не является (qr — мастер цены только предоплаченных; Baza цен живой продажи не знает и не должна). Класть финансы в org = тащить чужой домен в систему, недоступную на офлайн-КПП.
5. **Минимум точек отказа в горячем пути впуска** — очередь на КПП не ждёт. Меньше сетевых зависимостей = меньше риска срыва входа.

**Разграничение доменов** (принцип «где данные физически рождаются и где живут под защитой»):

- **Baza** — МАСТЕР факта прохода и оперативных операций смены (запись офлайн на месте): впуск, смена, состав/роли, живая продажа, матответственность, браслет↔билет, остатки.
- **org** — МАСТЕР каталога/заказа/билета + ПРИЁМНИК агрегированной финансово-аудиторской дельты, долгого хранения ПДн новых гостей живой продажи и реестра согласий 152-ФЗ.
- Шов = граница доступности сети: всё, что должно работать офлайн на КПП, физически лежит и пишется в Baza.
- **Запрет:** cross-schema JOIN из org в таблицы Baza (скрытая связь моделей, обход D14). Read-аналитика — только server-to-server read-API Baza или реплика.

**Разрешение главного расхождения (финансы — в Baza или в org).** Архитектор/DevOps: финансы в Baza (источник на КПП). Security/152-ФЗ: аудит/master в org (защищённый сервер, domain_history). Разрешаю **двухслойно**: операцию (приход/расход/инкассация/залог/живая продажа) **пишет Baza локально** (иначе офлайн не работает) как **append-only ленту**, которая синкается в org как **неперезаписываемый аудит + реестр согласий**. Удовлетворяет обе перспективы.

## 4. Стек UI

**Новый UI Baza — отдельное offline-first PWA (Vite + PrimeVue, единый визуальный язык с AdminFront), но самостоятельное приложение со своей session-аутентификацией, НЕ раздел AdminFront.**

- PWA даёт офлайн (service worker кэширует оболочку и ассеты локально — закрывает скрытый блокер: сейчас Baza-UI тянет vendor/QrScanner с `vhod.spaceofjoy.ru` по абсолютным URL, без локализации офлайн невозможен), install на телефон, IndexedDB под очередь операций.
- Камера (`getUserMedia`) требует secure context → локальный SSL (mkcert/локальный CA + локальный DNS/hosts) обязателен в любом варианте.
- **Оговорки (иначе скатимся в Вариант 2):** отдельное приложение + отдельная auth (общий только UI-китч/дизайн-система); лёгкий бандл (сканер/впуск/продажа открываются мгновенно); CI-job сборки Baza-PWA (не повторять TD-32); WebSocket-чат смены — НЕ несущий механизм (offline-first очередь, realtime только онлайн-бонус на штабе).

> Альтернатива (доработка Blade+jQuery до офлайна) проще по объёму, но структурно не offline-first (нет реактивного локального состояния и очереди операций под append-only финансы) — тупик под растущий функционал.

## 5. Интеграция org ↔ Baza (контракт по D14)

Два узких асинхронных канала + read-канал аналитики. **Никаких синхронных сетевых вызовов org из горячего пути впуска.**

**1) org → Baza: доставка билета (C11)**
- Основной путь: **ingest-API Baza** (`POST /api/baza/ingest/ticket`, `X-Baza-Token`, идемпотентно по UUID, явный версионированный DTO билета). Сейчас в `Baza/routes/api.php` только `/user` — эндпоинт новый.
- Fallback: прямая запись в БД `mysqlBaza` (как сейчас — `InMemoryMySqlTicketsRepository`, `TicketResponse::toArrayForBaza()/toArrayForSpisok()`). Срабатывает, только если ingest-API недоступно. Остаётся аварийным резервом.
- Транспорт: переиспользовать `Backend/src/BazaDelivery/` (очередь `baza_deliveries`, машина `queued→sending→delivered/failed`, ретрай [30,120,600], кап 10, история каждой попытки). Job маршрутизируется: ingest-API → fallback DB.

**2) Baza → org: вебхук «билет прошёл» (D14, новый)**
- Эндпоинт org `POST /api/v1/baza/ticketEntered` (S2S, по образцу `qr.ingest`). Контракт: `{ ticket_uuid, order_id?, festival_id, change_id, entered_at, target, wristband_qr? }`, идемпотентно.
- Действие org: запись в `domain_history` (`actor_type=baza`, `actor_id=null`). org только слушает, не командует Baza.
- Вызов из Baza при `enter` (после успешного впуска, под `Auth` PR #84). **Буферизуется в локальную очередь Baza** и доезжает при сети — не блокирует впуск.

**3) Baza → org: финансово-аудиторская дельта (новый, для разрешения спора о финансах)**
- ~~Append-only лента операций смены в org как аудит~~ — **ОТМЕНЕНО решением владельца №1 (§0):** финансы целиком в Baza, без синка аудита в org. Аудит/неперезаписываемость/защита ПДн — внутри Baza (append-only лента + soft-delete + шифрование диска ноутбука).

**4) Baza → org: read-аналитика (AF-2)**
- Только server-to-server (read-API Baza или реплика); AdminFront — под своим JWT. **Запрет cross-schema JOIN.** Аналитика помечается «данные на момент последней синхронизации».

Сквозные ID (UUID, `KEY_LIVE_TICKET`) — Published Language, не трогаем. `festival_id` (C12) — из жёсткой константы в управляемый контекст активного фестиваля внутри Baza. Формат ingest-API/вебхука — в общих `.claude/specs/` для cross-Claude (TD-24/Q9). Транспорт вебхуков — RabbitMQ с буферизацией.

## 6. Миграционный путь (без поломки текущего входа)

- **Шаг 0 (предусловие):** локализовать ассеты Baza-UI (относительные пути вместо `https://vhod.spaceofjoy.ru/`), убрать хардкод прод-домена. Не ломает вход.
- **Шаг 1:** безопасность входа — **сделано (PR #84)**. Доп.: закрыть `/users`, `/profile`, `/password-request` под `auth`; убрать тестовые пароли `secret` (до релиза финансовых ролей).
- **Шаг 2:** роли — таблица `change_user(change_id,user_id,role)` параллельно плоскому `changes.user_id` (двойная запись на переход). `is_admin` → 5 ролей мягким маппингом. Инвариант «у смены обязан быть главный» — мягко (валидация новых смен).
- **Шаг 3:** ingest-API Baza + маршрутизация `DeliverTicketToBazaJob` (ingest → fallback DB). Прямая запись `mysqlBaza` работает весь переход — доставка не прерывается.
- **Шаг 4:** вебхук Baza→org `POST /api/v1/baza/ticketEntered` + буферизованная очередь. Аддитивно — впуск работает и без него.
- **Шаг 5:** offline-first PWA Baza параллельно старому Blade (отдельный Docker-сервис `node-baza`, CI-job, локальный SSL mkcert + runbook). Старый Blade — fallback ≥1 фестиваль (Strangler). Cutover после паритета экранов и доказанного офлайна.
- **Шаг 6:** финансы/матответственность/браслеты с нуля — append-only лента операций + soft-delete смены + аудит + сущность `wristbands` + форма быстрой живой продажи (нал/безнал/перевод + данные гостя + согласие 152-ФЗ). Синк финансов append-only по `operation_id` (НЕ last-writer-wins).
- **Шаг 7:** read-витрина смен/финансов в AdminFront через server-to-server read-API/реплику (не JOIN). Помечать «на момент последней синхронизации».
- **Шаг 8 (security до выноса финансов+ПДн):** full-disk encryption ноутбука + минимизация ПДн в офлайн-выгрузке + удаление `baza.sql` из истории git + blacklist отменённых билетов в синке.
