## 1. Контекст и стратегическое решение

### 1.1 Новая роль org
Владелец зафиксировал разворот системы:
- **org (текущий SPA `FrontEnd/`) становится внутренней admin-only системой.** Доступ — только админ.
- **Публичная витрина и продажи билетов переезжают на отдельный сервис `qr.spaceofjoy.ru`** (отдельный репозиторий/Claude).
- Новая функция org: **создание билетов + контроль их доставки в систему `baza`** (baza — приложение аутентификации/хранилища билетов в этом же монорепо).

Следствие: публичные экраны org (покупка, анкеты, инвайты) больше не нужны в org — их роль уезжает на qr. org-фронт можно **пересобрать как чистую админку**, не таща легаси публичной части.

### 1.2 Решение по реализации (принято владельцем)
1. **Полный переезд фронта на Vite** (уход от Vue CLI 5 / webpack).
2. **Готовый admin-шаблон: PrimeVue Sakai** (Vite, **лицензия MIT** — подтверждено, свободно для закрытого/коммерческого использования).
3. **Greenfield в новом Docker-контейнере рядом** со старым `FrontEnd/`, без мутации старого webpack-проекта. Постепенный cutover (Strangler).

### 1.3 Почему Sakai, а не Vuetify / не готовый шаблон другого стека
- Мы **уже вложились в PrimeVue 4.5 (Aura, `cssLayer:true`)**: интегрирован в `main.js`, сделан рабочий экран `QrOrderListView.vue` (server-side DataTable + фильтры + Dialog + Timeline) + Vuex-модуль `QrOrderModule`.
- Sakai — это **тот же PrimeVue + Vite + Vue 3 + `<script setup>` + Pinia**. Экран qr-заказов и Vuex-модуль переезжают **1:1**, тема Aura сохраняется.
- Vuetify-шаблон (Materio/Sneat) выкинул бы PrimeVue и потребовал переписать готовый экран на Material Design — двойная работа ради «другого красивого старта». Отвергнут.

## 2. Целевая архитектура

### 2.1 Структура каталогов
```
systo/
├── FrontEnd/        ← старый Vue CLI/webpack. НЕ ТРОГАЕМ — держит прод до cutover.
├── AdminFront/      ← новый Vite + Sakai (чистая админка org)
└── docker-compose.yml
```

### 2.2 Стек нового фронта (`AdminFront/`)
| Слой | Выбор | Примечание |
|---|---|---|
| Сборка | **Vite** (latest stable) | основа `create-vue` (как Sakai) |
| Фреймворк | **Vue 3** (`<script setup>`) | как уже пишем новые экраны |
| UI | **PrimeVue 4.5 + тема Aura** | переносится из текущей интеграции |
| Layout | **Sakai** (топбар + сайдбар + breadcrumb + конфигуратор темы) | заменяет ручной `MenuView.vue` |
| Роутер | **vue-router 4** | `beforeEach` (auth+role) переносим 1:1 |
| Store | **Vuex 4** (домены) **+ Pinia** (только layout Sakai) | миграция Vuex→Pinia — ПОСЛЕ cutover, не блокирует |
| HTTP | **axios** + JWT-интерсепторы | копия логики `FrontEnd/src/main.js` |
| Иконки | **primeicons** | уже используем |

**НЕ переносим:** Bootstrap (грузился линками из `public/index.html`), jQuery, `v-calendar` (заменяем на PrimeVue DatePicker). Весь смысл greenfield — чистая админ-зона без легаси-стилей.

### 2.3 Новый Docker-сервис
Добавить в `docker-compose.yml` (существующие сервисы не менять):
```yaml
  node-admin:
    image: node:lts             # НЕ node:latest — детерминизм сборки
    container_name: node-admin-solarSysto
    working_dir: /app
    ports:
      - "5173:5173"             # Vite default; 8082 занят старым node — не конфликтуем
    volumes:
      - ./AdminFront:/app
    command: sh -c "npm run dev -- --host 0.0.0.0"
    environment:
      - VITE_API_URL=${VITE_API_URL:-http://api.tickets.loc/}
      - CHOKIDAR_USEPOLLING=true   # HMR на смонтированном bind-volume
    networks:
      - backend
```
Решения и обоснование:
- **`node:lts`** вместо `latest` — `latest` плавает по версиям, мина для сборки.
- **`--host 0.0.0.0`** — иначе Vite слушает только внутри контейнера, с хоста не достучаться.
- **`CHOKIDAR_USEPOLLING=true`** — на bind-volume inotify от хоста часто не доходит → HMR молчит. На Manjaro-хосте может работать и без него; оставляем как страховку.

### 2.4 Обход «задушенной сети» контейнера
У старого сервиса `node-solarSysto` **сломана/задушена сеть к npm registry** (npm install висит; пакеты для `FrontEnd/` ставили с хоста). Стратегия:
- **PoC (вариант A):** `npm install` гоняем **на хосте** в `AdminFront/` (хостовая сеть рабочая), контейнер монтирует всё (`./AdminFront:/app` включая `node_modules`) и только **запускает** Vite. Быстро, доказывает связку.
  - Риск: native-биндинги (esbuild/rollup) хост↔контейнер. Хост Linux x86_64 + `node:lts` Linux x86_64 → обычно совпадают. Если «wrong platform» — переходим на вариант B.
- **Целевое (вариант B):** named volume `admin_node_modules` поверх бинда, `npm install` разово внутри контейнера (если/когда сеть контейнера оживёт). Изоляция deps от хоста.
- Перед стартом — **проверить, воспроизводится ли вообще проблема сети на `node:lts`** (возможно, дело было в старом образе).

### 2.5 Перенос ядра — что 1:1, что переписать
| Кусок | Решение |
|---|---|
| vue-router 4 + `beforeEach` | **1:1**, но из 30+ роутов берём **только admin** (`requiresAuth + role`). Публичные не тащим. |
| axios + JWT-интерсепторы | **1:1** копия из `main.js`; меняется только env: `process.env.VUE_APP_API_URL` → `import.meta.env.VITE_API_URL`. |
| Vuex-модули | **копируем как есть**, в первый заход — только нужные admin-экранам (`appUser`, `appQrOrder`, `appFestivalTickets`, дальше по мере переноса). |
| env | `VUE_APP_*` → `VITE_*` (`import.meta.env.VITE_*`). |
| алиас `@/` | в `vite.config.js`: `resolve.alias { '@': fileURLToPath(new URL('./src', import.meta.url)) }`. |
| PrimeVue init | **1:1** (`use(PrimeVue, { theme: { preset: Aura, options: { cssLayer:true, ... }}})`). На чистой админке `cssLayer` уже не критичен (Bootstrap не грузим). |
| `QrOrderListView.vue` + `QrOrderModule` | **копипаст без изменений** — экран уже чистый PrimeVue. |

**Vuex vs Pinia:** оставляем **Vuex** для доменных модулей (16 рабочих, оттестированных жизнью), **Pinia** ставим только под layout-стор Sakai (тема/меню). Они мирно сосуществуют. Миграцию Vuex→Pinia — **постепенно, по модулю, после cutover**. Не блокируем переезд рефакторингом стора.

## 4. Cutover-стратегия (Strangler Fig)

Новый фронт растёт рядом со старым, постепенно «удушает» его, пока тот не станет не нужен. На каждом шаге **прод не ломается**, откат = вернуть Nginx-route на старый фронт.

**Фазы выкатки:**
1. **Параллельно, невидимо для прод-юзеров.** Старый `FrontEnd/dist` обслуживает всё; новый `AdminFront` — локально (`localhost:5173`) и на staging под отдельным хостом.
2. **Staging-выкатка нового фронта.** Маршрутизация — **вариант B: Nginx `/admin` → новый фронт, остальное → старый** (один origin). Это снимает разом **CORS** и **JWT-между-доменами** (токен в localStorage **не шарится между origin** — поддомен потребовал бы перелогин). Для этого: `base: '/admin/'` в `vite.config` + `createWebHistory('/admin/')`.
   - Альтернатива (отвергнута для старта): отдельный поддомен `admin.spaceofjoy.ru` → CORS-whitelist (TD-26) + проблема localStorage-токена между origin.
3. **Паритет по экранам.** Переносим Bootstrap-экраны по одному (заказы → анкеты → справочники → промокоды). Меню Sakai наполняется. Каждый экран = отдельный PR.
4. **Cutover.** Когда все admin-экраны перенесены и проверены — переключаем прод на новый `dist`. Старый `FrontEnd/` остаётся fallback ещё 1–2 фестиваля, потом удаляется.

## 5. PoC — первый осязаемый шаг

**Цель:** доказать связку `новый Docker + Vite + Vue3 + PrimeVue/Sakai + JWT-логин + реальный экран против живого API`.

**Состав (всё в `AdminFront/`):**
1. Sakai-стартер, **вычищенный до layout** (топбар + сайдбар + 1–2 пункта меню; демо-страницы `/uikit`, `/pages/*`, лендинг — удалить). Сохранить файл `LICENSE` (MIT).
2. `vite.config.js`: алиас `@/`, `server.host`, polling при нужде.
3. `main.js`: PrimeVue Aura + axios JWT-интерсепторы (копия из старого `main.js`, env → `import.meta.env`).
4. vue-router 4 с `beforeEach` + **2 роута**: `/login`, `/admin/qr-orders`.
5. Vuex: **3 модуля** — `appUser` (логин/refresh/isCorrectRole), `appQrOrder` (экран), `appFestivalTickets` (экран использует). Копируем папки как есть.
6. `QrOrderListView.vue` — **копипаст без изменений**.
7. Временный `LoginView` (получить токен в localStorage).
8. Docker: сервис `node-admin` (вариант A для node_modules).

**Критерии приёмки PoC (все 6 зелёные = связка доказана):**
1. `docker compose up node-admin` → `localhost:5173` отдаёт Sakai-layout.
2. HMR: правка `.vue` → обновление без перезагрузки.
3. Логин админом → токен в `localStorage['user.token']`.
4. `/admin/qr-orders` → DataTable подтянул **реальные данные** из `/api/v1/...` (JWT + CORS + server-side пагинация работают).
5. «Детали» → Dialog с гостями + Timeline истории.
6. Silent refresh: протухший токен → авто-refresh → запрос повторён.

## 7. Риски и митигации

| Риск | Серьёзность | Митигация |
|---|---|---|
| Два фронта в репо до cutover (дубль CI/конфигов) | Средний | Принять как цену greenfield; держать срок параллельной жизни коротким; отдельный CI-job `build-admin`. |
| Сеть контейнера снова задушит npm | Средний | Ставить deps на хосте (вариант A); проверить, воспроизводится ли на `node:lts`. |
| native-биндинги esbuild/rollup хост↔контейнер | Низкий–средний | Linux x86_64 = Linux x86_64 обычно ок; иначе вариант B (install внутри, named volume). |
| CORS на новый origin | Средний | **Обходим вариантом B** (route `/admin`, тот же origin). |
| JWT между доменами (localStorage не шарится) | Средний | **Обходим вариантом B** (один origin). |
| `base: '/admin/'` забыт → ассеты 404 | Низкий | `vite.config base` + `createWebHistory('/admin/')`. |
| Pinia + Vuex путают команду | Низкий | Правило: Pinia — только layout, данные — Vuex. Записать в README `AdminFront/`. |
| Перенос Bootstrap-экранов с jQuery-модалками | Средний | Основная трудоёмкость фазы 2–3; переписать модалки на PrimeVue Dialog. |
| «Доставка в baza» без бэкенда | Блокер фазы 4 | Согласовать эндпоинт sync-статуса + retry с backend заранее. |

## 8. Зафиксированные решения и открытые вопросы

### Зафиксировано
1. Полный переезд на **Vite**.
2. Шаблон — **PrimeVue Sakai** (MIT, подтверждено).
3. **Новый Docker** `node-admin` (greenfield), каталог **`AdminFront/`**.
4. **Vuex остаётся**, Pinia — только layout; миграция стора — после cutover.
5. Cutover — **Nginx `/admin`, один origin** (не поддомен).
6. node_modules для PoC — **вариант A** (install на хосте).

### Открытые (нужны до фазы 4 / по ходу)
1. **«Доставка в baza»**: есть ли уже бэкенд-эндпоинт статуса синхронизации билет→baza + ручной retry, или проектировать с нуля? (блокер фазы 4)
2. Где гоняем PoC — только локально, или сразу staging под отдельным хостом?
3. CI-job сборки `AdminFront/` — в PoC или позже?
4. Пред-существующий баг staging: все `auth:api`-роуты без токена отдают **500** (`Route [login] not defined` в `Authenticate.php`) — чинить отдельной веткой (не блокирует залогиненного админа, но мешает чистой 401).
