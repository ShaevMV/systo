# QR → ORG — публикация заказов через RabbitMQ (контракт для команды qr)

> **Что это:** контракт для команды/ИИ-агента витрины **qr.spaceofjoy.ru** — как публиковать заказы/статусы/письма в **RabbitMQ**, чтобы их забирал org. Дополняет существующий HTTP-канал (`qrOrder/create`, `changeStatus`, `emailNotification/send`), **не заменяя его тела сообщений** — JSON в сообщении такой же, как в HTTP-запросе.
> **Дата:** 2026-06-21. **Статус:** 🟡 **ПЛАН / черновик контракта** — брокерный канал ещё НЕ активирован в проде (см. §10). Сейчас живой канал — HTTP. Этот документ описывает целевую схему, по которой обе стороны будут готовиться.
> **Машиночитаемая версия:** `asyncapi.yaml` (в этом же каталоге).
> **Связанные документы:** `QR_CREATE_API.md` (тело заказа), `ORDER_STATUSES_LIFECYCLE.md` (статусы), `EMAIL_SEND_API.md` (S2S-письма), `QR_INGEST_AUTH.md` (текущий HTTP-токен).

---

## 0. Главное в двух абзацах

org переходит на приём заказов через **RabbitMQ как основной канал**, а HTTP S2S (`qrOrder/create` и пр.) остаётся **запасным/синхронным каналом** (решение владельца, 2026-06-21). Вам нужно начать **публиковать те же сообщения** в брокер. **Тело сообщения = ровно тот же JSON**, что вы сейчас шлёте по HTTP — менять формат данных не нужно, меняется только транспорт (AMQP вместо HTTPS).

**Идемпотентность не меняется:** ключ — `order_id` (== id заказа в org). Дубль (ретрай публикации, повтор доставки брокером, или сообщение, продублированное в HTTP *и* в AMQP) org отсекает по `order_id` — повторной записи и повторной выдачи билетов не будет. Поэтому публиковать безопасно даже «с запасом».

## 1. Модель интеграции через брокер

| | |
|---|---|
| **Роль qr** | **Продьюсер** — публикует сообщения в exchange `x.qr.inbound`. Ничего из брокера не читает. |
| **Роль org** | **Консьюмер** — забирает сообщения и прогоняет через тот же пайплайн выдачи, что и HTTP (создание заказа → билеты → PDF → письма → запись в Baza). |
| **Транспорт** | AMQP **только поверх TLS** (AMQPS, порт `5671`) со **взаимной аутентификацией (mTLS)** — см. §2. Открытый `5672` наружу закрыт. |
| **Тело сообщения** | JSON, **идентичный** соответствующему HTTP-контракту (см. §4). org нормализует/валидирует так же, как на HTTP. |
| **Идентификатор** | `order_id` — общий, id qr == id org. Маппинга нет. |
| **Расчёт цены** | Делает qr. org принимает `price.total`/`payment.amount_total` как есть (как и по HTTP). |

> **Почему брокер, а не только HTTP:** (1) **буфер** — если org на деплое/недоступен, сообщения копятся в durable-очереди, не теряются; (2) **развязка** — publish-and-forget, не ждёте синхронной выдачи билетов; (3) **наблюдаемость** — поток заказов виден в Grafana (глубина очередей, лаг, застрявшие).

## 2. Подключение (mTLS)

Канал защищён **взаимным TLS**: и сервер (брокер), и клиент (qr) предъявляют сертификаты. Это сильнее логина-пароля и привязывает доступ к **вашему** сертификату.

| Параметр | Значение | Примечание |
|---|---|---|
| Протокол | `amqps://` (AMQP over TLS) | НЕ `amqp://` |
| Хост | **TBD** — `mq.spaceofjoy.ru` (рабочее имя, уточняется) | Отдельный домен брокера (§10) |
| Порт | `5671` | TLS-листенер |
| vhost | `qr-integration` | Отдельный vhost под интеграцию (изоляция от внутренних очередей org) |
| Пользователь | `qr_ingest` | Права **только на publish** в `x.qr.inbound` (`configure=^$`, `write=^x\.qr\.inbound$`, `read=^$`) |
| Пароль | выдаётся отдельно (секрет) | Поверх mTLS — второй фактор |
| Client cert | **выдаст org** (ваш клиентский сертификат + ключ) | Предъявляется при TLS-handshake (`verify_peer`, `fail_if_no_peer_cert=true`) |
| CA cert | **выдаст org** | Для проверки серверного сертификата брокера |

**Что вам понадобится от org перед подключением:** `ca.pem` (корневой), `qr-client.pem` + `qr-client.key` (ваш клиентский серт/ключ), логин/пароль `qr_ingest`, финальный хост:порт. Это передаётся защищённым каналом (не в открытом виде, не в git).

**Рекомендации клиента (любой стандартный AMQP-клиент, напр. `pika`/`aio-pika` для Python):**
- `publisher confirms` **включены** — публиковать с подтверждением (`confirm_delivery`), считать сообщение принятым только после `ack` от брокера. Без этого возможна тихая потеря при разрыве.
- `delivery_mode=2` (persistent) на каждом сообщении — переживает рестарт брокера.
- Реконнект с backoff при разрыве TLS/heartbeat.
- TLS-проверка хоста включена (не отключать verify).

## 3. Топология и конверт сообщения

```
publish → exchange  x.qr.inbound   (type: topic, durable)
            ├─ routing key: qr.order.create   → создание заказа
            ├─ routing key: qr.order.status   → смена статуса
            └─ routing key: qr.email.send     → S2S-письмо (не-заказное)
```

Очереди, DLQ, ретраи — **на стороне org** (вам их объявлять не нужно; вы только публикуете в `x.qr.inbound` с нужным routing key).

**Свойства/заголовки каждого сообщения:**

| Свойство (AMQP) | Значение | Зачем |
|---|---|---|
| `content_type` | `application/json` | — |
| `delivery_mode` | `2` (persistent) | durability |
| **`message_id`** | **`order_id`** (для писем — `external_id`) | **ключ идемпотентности и дедупликации** |
| `type` (или header `x-message-type`) | `qr.order.create` / `qr.order.status` / `qr.email.send` | роутинг внутри консьюмера org |
| header `x-schema-version` | `1` | версия контракта; незнакомую версию org уводит в DLQ, не падая |
| `correlation_id` | `order_id` (рекомендуется) | сквозная трассировка в логах/Grafana |
| `timestamp` | время публикации | наблюдаемость |

## 4. Что публиковать (3 типа сообщений)

**Тела — без изменений относительно HTTP.** Берёте ровно тот JSON, что слали бы в соответствующий HTTP-эндпоинт, и кладёте телом AMQP-сообщения.

### 4.1. Создание заказа — `routing key = qr.order.create`
- Тело = тело `POST /api/v1/qrOrder/create` (см. **`QR_CREATE_API.md`** / `QR_FULL_EXAMPLE.md`).
- `message_id = order_id`.
- Поведение org — как при HTTP-приёме: `status="создан"` → письмо «заказ создан» (для `regular`), `status="оплачен"` → немедленная выдача билетов (один раз).

### 4.2. Смена статуса — `routing key = qr.order.status`
- Тело = `{ "order_id": "<uuid>", "status": "оплачен" }` (то же, что `POST /qrOrder/changeStatus/{id}`, но `order_id` кладём **в тело**, т.к. в брокере нет URL-пути).
- `message_id = order_id` (тот же order_id — это нормально; см. ниже про дедуп смены статуса).
- Поведение — как HTTP: переход в «оплачен» при `issued_at == null` запускает выдачу один раз.

### 4.3. S2S-письмо (не-заказное) — `routing key = qr.email.send`
- Тело = тело `POST /api/v1/emailNotification/send` (см. **`EMAIL_SEND_API.md`**): `{ event, email, vars{}, festival_id?, order_type?, ticket_type_id?, subject?, aggregate_id?, external_id? }`.
- `message_id = external_id` (ключ идемпотентности писем).
- Только не-заказные события (`user_registered`, `password_reset`, `invite`, `questionnaire`, `questionnaire_approved`, `questionnaire_rejected`). Заказные письма org формирует сам — не дублировать.

> **Нюанс дедупа для смены статуса:** идемпотентность выдачи защищена `issued_at` на стороне org (повторный «оплачен» билеты второй раз не выпустит). Если хотите различать события «создан»/«оплачен» по одному `order_id` на уровне брокера — можно добавить в `message_id` суффикс статуса (напр. `<order_id>:paid`), но это **опционально**: даже без него повтор безопасен.

## 5. Обработка ошибок и ретраи (что делает org, что делаете вы)

org реализует ту же развилку, что и HTTP-коды `422` vs `5xx`:

| Ситуация на стороне org | Что делает org | Что делаете вы (qr) |
|---|---|---|
| **Битый контракт** (нет обязательных полей, невалидный uuid, >1000 гостей) — аналог HTTP `422` | Сообщение уходит в **DLQ** (`q.qr.dlq`), НЕ ретраится бесконечно. Админ org видит его в UI/дашборде | Ничего; при необходимости org свяжется. Можно дослать исправленное новым сообщением |
| **Транзиентный сбой** (БД недоступна, временная 5xx-ситуация) | org сам делает **nack + отложенный ретрай** (30/120/600 сек), затем — DLQ | **Ничего не делать** — не публиковать повторно вручную, org доедет ретраем |
| **Неизвестная `x-schema-version`** | DLQ | Согласовать версию контракта |
| **Успех** | `ack`, заказ обработан | publisher confirm подтверждён |

**Вывод для вас:** публикуете один раз с publisher-confirm. Если брокер подтвердил приём (`ack`) — дальше доставка и ретраи на org. Дублировать публикацию для надёжности не нужно (но и не вредно — `order_id` отсечёт дубль).

## 6. HTTP как запасной канал (fallback)

Пока брокер — основной, **HTTP-канал остаётся рабочим**. Сценарии fallback:
- Брокер недоступен (TLS не поднимается, очередь не принимает) → можете дослать тот же заказ по HTTP (`qrOrder/create`) — `order_id` гарантирует отсутствие дубля.
- Нужен **синхронный** ответ (сразу узнать `422` по битому контракту при отладке) → HTTP удобнее: код ответа приходит сразу, а не «улетает в DLQ».

То есть оба канала ведут в **одну и ту же** идемпотентную точку приёма org. Это безопасно by design.

## 7. Чего НЕ делать

- ❌ Не подключаться по `amqp://` (открытый, без TLS) — только `amqps://` (mTLS).
- ❌ Не объявлять/не удалять очереди и exchange самим — у `qr_ingest` прав только publish. Топологию держит org.
- ❌ Не публиковать заказные письма (`order_*`, `list_*`) через `qr.email.send` — org шлёт их сам (будет дубль).
- ❌ Не менять формат тела ради брокера — он идентичен HTTP-контракту.
- ❌ Не отключать publisher confirms / TLS verify «чтобы быстрее заработало».

---

## 8. Порядок внедрения (рекомендованный, обратносовместимый)

1. **Shadow** — публикуете в брокер **параллельно** текущему HTTP. org сверяет, что AMQP-путь даёт идентичный результат (по истории заказа/дашбордам). Риска нет: `order_id` отсекает дубли.
2. **Cutover** — переключаетесь на «AMQP — основной, HTTP — fallback при недоступности брокера».
3. HTTP остаётся в строю как страховка (минимум ещё один фестиваль).

## 9. Открытые вопросы / TBD (уточняются org-стороной)

- **Хост брокера** (`mq.spaceofjoy.ru` — рабочее имя) — финальный домен/IP зависит от выбора отдельного сервера. Будет сообщён.
- **IP allowlist** — если у qr статичный исходящий IP, org добавит его в firewall на `5671` (второй барьер поверх mTLS). Пришлите ваш IP/подсеть.
- **Версионирование схемы** — `x-schema-version: 1` сейчас. Изменения контракта — через bump версии + период двойной поддержки.
- **Прод vs staging** — на staging брокер уже есть, но mTLS-листенер ещё не активирован; на проде брокера пока нет вовсе. Активируем по готовности (см. roadmap v2.7.0).
