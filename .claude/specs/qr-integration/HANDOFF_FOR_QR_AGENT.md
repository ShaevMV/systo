Привет! Собрали для вас всё по статусам заказа и письмам. Ниже — готовая шпаргалка.

## 1. Статусы заказа, которые мы принимаем

Статус передаётся в двух местах:
- при приёме заказа — поле `order_data.status` в `POST /api/v1/qrOrder/create`;
- при смене статуса — тело `{ "status": "..." }` в `POST /api/v1/qrOrder/changeStatus/{order_id}`.

**Регистр и пробелы не важны** — мы нормализуем значение сами (`Оплачен `, `ОПЛАЧЕН`, `оплачен` — одно и то же). Русские строки предпочтительны, английские синонимы тоже принимаются.

| Что слать | Синонимы | Что делает org |
|---|---|---|
| **`создан`** | `new`, `created` | Сохраняет заказ. Шлёт письмо **«заказ создан»** (шаблон `orderToCreate`) на `order_data.email` — **только для `type_order = regular`**. Билеты **НЕ** выпускает, `issued_at` остаётся пустым. |
| **`оплачен`** | `paid` | Запускает **выдачу билетов один раз**: создание билетов → PDF → письмо **«оплачен»** с билетами → запись в Baza (вход) → Telegram. Набор шагов зависит от `type_order`. |
| **прочее** (например `отменён`) | — | Только сохраняется и пишется в историю заказа (`status_changed`). **Действий с билетами / отмены билетов пока НЕТ** (отмена через qr запланирована, но ещё не реализована). |

## 2. Тип заказа (`order_data.type_order`)

Допустимые значения: `regular` / `friendly` / `list` / `live`. Пустое или неизвестное значение → трактуется как `regular`.

Нюанс по письму «заказ создан»: оно отправляется **только для `regular`**. Причина:
- **`list`** (заказы-списки) — при создании письма не шлют (по дизайну);
- **`live`** (живые билеты) — стартуют сразу оплаченными, отдельного шага «создан» для них нет.

## 3. Жизненный цикл — два режима (выбирайте любой)

**Режим A — двухшаговый** (рекомендуется, если на витрине есть отдельный шаг оплаты):
1. `create` со статусом `создан` → мы шлём письмо «заказ создан», билеты не выпускаем.
2. `changeStatus` со статусом `оплачен` → мы выпускаем билеты (PDF + письмо «оплачен» + Baza + Telegram).

**Режим B — одношаговый** (если заказ приходит уже оплаченным):
- `create` сразу со статусом `оплачен` → мы немедленно выпускаем билеты. `changeStatus` не нужен, письма «создан» при этом не будет.

### Эндпоинт смены статуса (восстановлен)

```
POST /api/v1/qrOrder/changeStatus/{order_id}
Заголовок: X-QR-Token: <тот же ключ, что у qrOrder/create>
Тело:      { "status": "оплачен" }
```

`order_id` в пути = тот же `order_id`, что слался при `create` (id заказа qr == id заказа org).

**Идемпотентность выдачи:** выдача привязана к отметке `issued_at` — первый «оплачен» её ставит до запуска, поэтому повторный «оплачен» (ретрай, дубль вебхука) билеты второй раз **не выпустит**. Слать «оплачен» повторно безопасно.

Ответы `changeStatus`:

| Код | Тело | Что делать |
|---|---|---|
| `200` | `{ "success": true, "item": {...}, "message": "Статус обновлён" }` | успех (при «оплачен» — выдача запущена) |
| `422` | `{ "success": false, "message": "Не передан status" }` | исправить тело |
| `404` | `{ "success": false, "message": "Заказ не найден" }` | сначала сделать `qrOrder/create` |
| `401` | `{ "success": false, "message": "Доступ запрещён: ..." }` | **не ретраить** — проверить `X-QR-Token` |

## 4. Новый тип письма — «анкета одобрена»

В наш каталог писем добавлено новое событие:

| `event` | Дефолтный шаблон | Метка |
|---|---|---|
| `questionnaire_approved` | `questionnaireApproved` | Анкета одобрена |

- **Когда шлём сами:** при одобрении анкеты администратором в org гостю автоматически уходит письмо «анкета одобрена» — вам ничего слать не нужно.
- **Если хотите инициировать его сами** (например, анкета одобряется на вашей стороне) — через S2S-канал писем `POST /api/v1/emailNotification/send` (тот же `X-QR-Token`):
  - `event = "questionnaire_approved"`,
  - `email` — получатель,
  - `vars` — как минимум `{ "link": "..." }`,
  - рекомендуется `external_id` для идемпотентности.

## 5. Полный каталог событий писем (16)

Эти коды можно инициировать через `POST /api/v1/emailNotification/send`:

```
order_created
order_paid
order_paid_friendly
order_paid_live
order_cancel
order_changed
order_difficulties
order_live_issued
list_approved
list_cancel
list_difficulties
user_registered
password_reset
invite
questionnaire
questionnaire_approved
```

**Важно:** заказные/листовые письма (`order_*`, `list_*`) org формирует **сам** при приёме заказа и при сменах статуса — дублировать их через `emailNotification/send` не нужно (будет дубль). Для этого канала вам нужны прежде всего не-заказные события: `user_registered`, `password_reset`, `invite`, `questionnaire`, `questionnaire_approved`. Подробнее — в `EMAIL_SEND_API.md`.
