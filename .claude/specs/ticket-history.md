# Спецификация: История билета

> Кандидат в backlog. Не готов к разработке — требует уточнений (см. §9 Открытые вопросы).
> Документ предназначен для встречи с tech-lead и пользователем для согласования scope.

---

## 1. Бизнес-ценность

### Какую проблему решаем

Сейчас в системе есть **история заказа** (`OrderTicket` через `domain_history`), но **нет истории отдельного билета**. Это создаёт ряд бизнес-проблем:

1. **Невозможно ответить на вопрос «что было с конкретным билетом»** — пользователь обратился в поддержку с QR-кодом, но мы не видим:
   - когда билет был фактически выдан и кому отправлен
   - в какую смену сканер пропустил его на фестивале
   - менялись ли данные владельца билета (имя/email/телефон)
   - был ли билет когда-нибудь отозван или передан другому человеку

2. **Нет аудита смены владельца** — фича смены владельца только планируется (см. ниже), но без истории невозможно понять кто, когда и кому передал билет. Это критично для:
   - **разрешения споров** («это мой билет!» — «нет, он был передан 3 дня назад»)
   - **защиты от мошенничества** (массовая перепродажа билетов с накруткой цены)
   - **аналитики после фестиваля** (какая доля билетов сменила владельца? в какой момент?)

3. **Невозможна точная диагностика инцидентов на фестивале** — если на воротах возник конфликт («этот билет уже использован»), сейчас можно посмотреть только последнее значение `change_id` в Baza. Нет ответа на «кто пропустил билет 25 минут назад в какую смену и в каком статусе он находился до этого».

4. **Sync Backend → Baza — чёрный ящик** — `setInBaza()` тихо записывает билеты в Baza, но если что-то пошло не так (например, билет был отправлен, но в Baza по какой-то причине его нет), мы это узнаём только в момент сканирования на фестивале. Запись события `ticket.sent_to_baza` в историю даст:
   - точный timestamp успешного sync
   - возможность пакетной пересинхронизации (повторить все события за период)
   - наблюдаемость для devops (Sentry + Grafana — см. v2.7.0)

### Целевая аудитория

| Роль | Что получает |
|------|--------------|
| **Admin** | Полная история билета в админке — все события с timestamp и actor (UI рядом с историей заказа) |
| **Сканеры на воротах (seller)** | Возможность увидеть «уже пропускался ли этот билет в эту смену» прямо в Baza UI |
| **Куратор** | История своих билетов-списков (когда был отправлен, кто пропустил) |
| **Оргкомитет фестиваля** | После фестиваля — отчёт «билеты, по которым были аномалии» (отзыв, смена владельца, повторный пропуск) |
| **Гость / владелец билета** | В будущем (не в первой версии) — личный кабинет «история моего билета» (если есть email-привязка) |

### Метрики успеха

| Метрика | Сейчас | Цель после фичи | Как измерить |
|---------|--------|-----------------|--------------|
| Среднее время ответа на тикет «что с моим билетом» | ~15 минут (ручной поиск в БД) | < 1 минуты (один экран) | Замер по support-тикетам в Telegram |
| Доля билетов с зафиксированным фактом пропуска | ~0% (есть только последний `change_id`) | 100% (каждый скан — событие в истории) | Запрос `COUNT(domain_history WHERE event = ticket.scanned)` |
| Кол-во инцидентов «билет уже использован» без разрешения | Точно неизвестно | 0 (всегда есть кто и когда) | Самоотчёт сканеров после фестиваля |
| Время на расследование спора о владельце | Не было фичи | < 5 минут | По логам |

### Что будет, если не делать

- **При появлении массовой смены владельцев билетов** (а это произойдёт — пользователь сам поднял тему) — мы окажемся без аудита и любой спор будет «слово против слова»
- **При проблемах с sync Backend → Baza** — продолжим узнавать о них постфактум на воротах фестиваля
- **При вопросе оргкомитета «дайте отчёт по подозрительным билетам»** — придётся писать ручные SQL-запросы каждый раз

## 2. Сценарии использования (User Stories)

### US-1. Admin расследует жалобу владельца

**Как** admin,
**я хочу** открыть карточку билета и увидеть всю его историю в хронологическом порядке,
**чтобы** ответить пользователю «билет был создан 2026-04-10, отправлен в Baza 2026-04-10, передан другому владельцу 2026-04-15, пропущен на воротах 2026-04-20 в смену №3».

**Acceptance Criteria:**
- В админке открыт раздел «История» рядом с историей заказа
- Каждое событие имеет: timestamp, тип события, actor (кто инициировал), полезную нагрузку (например, новый владелец)
- События отсортированы по `occurred_at DESC`

### US-2. Куратор подтверждает что его список дошёл до Baza

**Как** куратор заказа-списка,
**я хочу** видеть статус «билеты переданы в систему сканирования» по каждому билету моего списка,
**чтобы** убедиться что в день фестиваля гостей пропустят без проблем.

**Acceptance Criteria:**
- В личном кабинете куратора (`/curatorOrders`) рядом с каждым билетом — иконка/метка «в Baza ✓» если в истории есть событие `ticket.sent_to_baza`
- Если события нет более 1 часа после `APPROVE_LIST` — alert куратору (необязательно для MVP)

### US-3. Сканер видит дубликат пропуска

**Как** сканер на воротах,
**я хочу** при сканировании QR увидеть «этот билет уже был пропущен 14:32 в смену №2 пользователем Иваном»,
**чтобы** не пропустить дубликат и обратиться к гостю.

**Acceptance Criteria:**
- При scan'е billete в Baza UI показывает не просто «уже использован», а детали: когда, кем, в какую смену
- Эту информацию Baza получает локально (история живёт в Baza или копия в Baza)
- В историю билета в Backend пишется событие `ticket.scan_attempted` (даже если повторный — для аудита попыток)

### US-4. Admin меняет владельца билета

**Как** admin,
**я хочу** заменить ФИО/email/телефон владельца билета через UI,
**чтобы** при сканировании на воротах был указан новый владелец, а у пользователя осталась запись о смене.

**Acceptance Criteria:**
- В админке есть кнопка «Сменить владельца» в карточке билета
- Форма: новое ФИО, email, телефон, причина (свободный текст), кому уведомить (старому/новому/обоим)
- После сохранения:
  - В историю билета пишется событие `ticket.owner_changed` с `from`/`to` и причиной
  - В Baza ретранслируется новое имя через `setInBaza()` (на воротах будет уже новое имя)
  - PDF билета **перегенерируется** с новым именем (старый PDF становится недействительным)
  - Email уведомление (по конфигурации) старому и новому владельцу
- Сменить владельца можно только если билет в статусе `PAID` / `LIVE_TICKET_ISSUED` (не для `CANCEL`)

### US-5. Возврат/отзыв билета с историей

**Как** admin,
**я хочу** отозвать билет (например, после смены статуса заказа на `CANCEL`),
**чтобы** в истории билета было видно «билет отозван 2026-04-25, причина: возврат денег».

**Acceptance Criteria:**
- При смене статуса заказа в `CANCEL` каждый билет получает событие `ticket.revoked` (`reason: order_cancelled`)
- При ручной отмене билета (отдельная фича — если будет) — событие `ticket.revoked` с другой причиной
- После отзыва — попытка скана на воротах вернёт «билет отозван» (синхронизация в Baza)

### US-6. Аналитика для оргкомитета

**Как** организатор фестиваля,
**я хочу** после фестиваля получить отчёт «билеты, которые меняли владельца более одного раза» и «билеты, по которым были попытки повторного сканирования»,
**чтобы** проанализировать возможные злоупотребления (перепродажа, мошенничество).

**Acceptance Criteria:**
- В админке доступен экспорт (CSV) с фильтром по фестивалю
- Колонки: ID билета, email текущего владельца, кол-во смен владельца, кол-во scan-попыток, ID последней смены
- Этот сценарий — *nice to have*, можно отложить на v2.8.0+

## 3. События истории билета

### Полный набор событий

| Event | Откуда пишется | Aggregate | Payload | Когда |
|-------|----------------|-----------|---------|-------|
| `ticket.created` | Backend (Ticket::newTicket) | `ticket` | `name`, `email`, `kilter`, `order_id`, `ticket_type_id` | При создании билета через `ProcessCreateTicket` |
| `ticket.qr_generated` | Backend (после `ProcessCreatingQRCode`) | `ticket` | `pdf_path` | После генерации PDF + QR |
| `ticket.sent_to_baza` | Backend (после `setInBaza`) | `ticket` | `baza_table` (`el_tickets`/`spisok_tickets`/`live_tickets`/...), `result` (success/fail), `attempt` (int) | При успешной/неуспешной записи в Baza |
| `ticket.owner_changed` | Backend (новая команда `ChangeTicketOwner`) | `ticket` | `from: {name, email, phone}`, `to: {name, email, phone}`, `reason`, `actor_id`, `notify_old`, `notify_new` | При смене владельца через admin UI |
| `ticket.data_changed` | Backend (`ChangeTicket` — уже существует, расширить) | `ticket` | `changes: {field: {from, to}}` | При редактировании данных билета без смены владельца |
| `ticket.email_sent` | Backend (после отправки писем с PDF) | `ticket` | `recipient`, `template`, `mailable_class` | Каждое письмо с билетом → ивент |
| `ticket.scanned_at_festival` | Baza → Backend (REST push) | `ticket` | `change_id`, `change_user_ids`, `baza_kilter`, `scanned_at`, `device_id?` | При успешном пропуске сканером |
| `ticket.scan_attempted_duplicate` | Baza → Backend (REST push) | `ticket` | `change_id`, `change_user_ids`, `previous_scan_at` | При попытке повторного сканирования (дубль) |
| `ticket.revoked` | Backend (при `CANCEL` заказа или ручной отмене) | `ticket` | `reason` (`order_cancelled`/`manual`/`refund`), `actor_id` | При отзыве билета |
| `ticket.live_number_assigned` | Backend (при `LIVE_TICKET_ISSUED`) | `ticket` | `live_number`, `seller_id` | При выдаче живого билета |

### Связь между ID

Билет в Backend — `Uuid`. Билет в Baza — `int` (`kilter`/`id`) в разных таблицах (`el_tickets`, `live_tickets`, `spisok_tickets`, `friendly_tickets`, `parking_tickets`). В `el_tickets` есть колонка `uuid`, в других таблицах — `el_ticket_id` или `ticket_uuid`.

**Для маппинга Baza → Backend** нужна резолверная таблица или поиск по uuid в каждой Baza-таблице. Это — отдельный архитектурный вопрос (см. §5).

## 4. Архитектурный подход (выбор)

### Вариант A: общий `domain_history`

Используем существующую таблицу `domain_history` с агрегатом `ticket` (по аналогии с `order`).

**Плюсы:**
- Не нужно создавать новую таблицу и миграции
- Уже есть инфраструктура: `HasHistory` trait, `ActorType`, репозиторий `HistoryRepositoryInterface`, контроллер `getHistory`
- Единый источник правды по всем событиям системы — удобно для будущего audit-flow (см. v2.7.0)
- Запрос «вся история всех агрегатов по этому пользователю» одним SQL
- Бизнес-сценарий «один билет — много событий разных aggregate type связаны (заказ → билет → анкета)» в одной таблице

**Минусы:**
- Таблица будет расти быстрее — каждый билет даст 5-10 событий + каждое сканирование на фестивале. Прогноз: 10 000 заказов/год × ~3 билета × ~8 событий = ~240 000 строк/год. Управляемо, но требует индексов на `(aggregate_type, aggregate_id, occurred_at)`.
- При больших объёмах партиционирование по `aggregate_type` будет неудобно (одна большая таблица)
- События пропуска билета на фестивале — самый «горячий» write-path, и они смешиваются с прохладными event'ами заказа

### Вариант B: отдельная таблица `ticket_history`

Создаём отдельную таблицу со схожей структурой, специально для билетов.

**Плюсы:**
- Изоляция write-path для сканирования (можно даже отдельный коннект к БД, если потребуется)
- Возможность отдельной retention policy (например, события сканирования храним 1 год, остальные навсегда)
- Чище для аналитики

**Минусы:**
- Дублирование инфраструктуры: `TicketHistoryRepositoryInterface`, отдельный CRUD, отдельный контроллер
- Разрыв единого источника правды
- Если в будущем добавим `questionnaire_history` и `payment_history` — получим зоопарк таблиц
- Усложнение UI «вся история по пользователю»

### Рекомендация бизнес-аналитика

**Вариант A (общий `domain_history`)**, потому что:

1. **Уже есть рабочая инфраструктура.** Не нужно изобретать колесо — `HasHistory` trait готов к подключению в `Ticket`.
2. **Объём событий управляем.** Текущая нагрузка — 10K заказов/год, 2 фестиваля. Даже после фестиваля при пиковой нагрузке (30K событий за выходные при сканировании всех билетов) — это не проблема для MySQL с правильными индексами.
3. **Единая аналитика.** Пользователь явно сказал в контексте про **v2.7.0 «audit + воронка покупки»** — там этот же `domain_history` будет хабом. Сделать билет частью этого хаба = логичная преемственность.
4. **Если в будущем будет нужно** — можно вынести в отдельную таблицу через горизонтальное масштабирование (read replica) или миграцию. Это легче чем сейчас угадывать.

**Рекомендация tech-lead'у:**
- Использовать вариант A
- Добавить миграцию: индекс `(aggregate_type, aggregate_id)` если его ещё нет
- В v2.7.0 при внедрении Grafana + Loki — мониторить рост `domain_history` и при необходимости перейти на вариант B

## 5. Связь Baza ↔ Backend (для событий сканирования)

Это **самый чувствительный архитектурный вопрос** — Baza сейчас работает изолированно (сканер на воротах, локальный ноутбук с сервером Baza, всё в офлайне) и пишет историю сканирования в свою БД.

### Текущее поведение

- `EnterTicket::skip()` вызывает `repository->skip($id, $changeId)` — пишет `change_id` и `date_change` в строку билета (`spisok_tickets`/`el_tickets`/...) с **перезаписью** последнего значения. Истории как таковой нет — только последний пропуск.
- Параллельно `AddTicketsInReport::increment($changeId, $typeTicket)` инкрементит счётчик в `changes`.

### Варианты ретрансляции в Backend

#### Опция 1: REST API push Baza → Backend (после каждого скана)

Baza после `skip()` делает POST на `https://api.spaceofjoy.ru/api/v1/ticket/scanned` с payload (`uuid`, `change_id`, `change_user_ids`, `scanned_at`).

**Плюсы:**
- Простота, понятный синхронный flow
- Backend получает событие сразу, история в `domain_history` свежая

**Минусы:**
- **Главный риск:** Baza на фестивале работает в офлайне (ноутбук-сканер с локальным сервером — см. roadmap v2.6.0 «SSL для ноутбука-сканера»). Если интернета нет, push провалится.
- Нужна обработка ошибок и retry queue в Baza (не реализовано)
- Скорость работы сканера зависит от ответа Backend (даже маленький лаг = проблема)

#### Опция 2: Webhook (асинхронный push с очередью в Baza)

В Baza локальная очередь скан-событий → фоновый воркер пушит в Backend, когда есть интернет.

**Плюсы:**
- Не блокирует сканер
- Устойчиво к офлайну
- Можно дать гарантию «every-event-eventually»

**Минусы:**
- Нужна очередь в Baza (новая инфраструктура — пока в Baza нет database-queue)
- Сложнее отладка, события могут приходить в Backend с большой задержкой
- Для US-3 (сканер видит «дубликат + кто пропустил») — события всё равно нужны локально в Baza, иначе Baza-side не работает

#### Опция 3: RabbitMQ (уже используется в Backend)

Baza публикует событие в очередь, Backend consumes.

**Плюсы:**
- В Backend уже работает RabbitMQ (см. `DOMAIN.md`)
- Единая шина событий, удобно для будущих интеграций

**Минусы:**
- RabbitMQ в Baza не настроен
- Та же проблема офлайн — что и с REST
- Overkill для одного типа событий

#### Опция 4: Батч-синхронизация (Baza pull от Backend)

Backend периодически (каждые 5 минут / по cron / при подключении интернета) запрашивает у Baza «дай все события сканирования с момента X» через GET /api/sync/scan-events.

**Плюсы:**
- Простота — нет push-логики в Baza
- Хорошо работает в офлайне (по факту восстановления интернета можно догнать)
- Минимум новой инфраструктуры

**Минусы:**
- Не real-time (история в Backend отстаёт на 5+ минут)
- Нужен endpoint в Baza с фильтром по timestamp
- Backend инициирует, что нетипично для нашей архитектуры

#### Опция 5: Гибрид (Baza ведёт локальную историю + батч-синк)

В Baza создаём свою локальную таблицу `ticket_scan_history` (полноценная история, не просто `change_id`). Это сразу даёт нам US-3 (сканер видит дубликат локально). Параллельно — батч-синк в Backend (опция 4) когда есть интернет.

**Плюсы:**
- US-3 решается полностью локально в Baza (важно для офлайна)
- Backend получает историю с допустимой задержкой
- Расширяемо: можно потом перейти на push если потребуется
- Минимум зависимости от интернета на фестивале

**Минусы:**
- Две таблицы истории (одна в Baza, одна в Backend)
- Нужна логика дедупликации при синке (по `scan_uuid`, генерируемом в Baza)

### Рекомендация бизнес-аналитика

**Опция 5 (Гибрид)** — единственный вариант который покрывает офлайн-сценарий (критичный — на фестивалях бывают проблемы со связью).

Реализация поэтапно:
- **Этап 1 (MVP)**: в Baza создаём `ticket_scan_history` (локальная история со всеми пропусками), плюс модифицируем `skip()` чтобы вместо перезаписи `change_id` писать новую запись в эту таблицу. На UI Baza сразу появляется «этот билет уже пропускался Х раз».
- **Этап 2**: добавляем endpoint в Baza `GET /api/sync/scan-events?since=...` (защищён `auth-token` как уже сделано в `bot` middleware Backend'а).
- **Этап 3**: в Backend добавляем cron-команду `php artisan ticket:sync-scan-history` (каждые 5 минут или по запросу). Команда читает batch, пишет в `domain_history`, помечает `scan_uuid` как обработанные.

**На случай если требуется real-time** (опция 1) — добавить в Этап 4: если интернет есть, Baza пушит сразу + батч как fallback.

## 6. Смена владельца билета — вопросы для уточнения

Это **самая сложная часть фичи** с точки зрения бизнес-правил. Нужны решения от пользователя.

### Q1. Кто может менять владельца?

**Варианты:**
- A) Только `admin`
- B) `admin` + `manager`
- C) `admin` + сам владелец билета (если авторизован)
- D) `admin` + куратор (для билетов из своего списка)

**Рекомендация:** начать с A (только admin), расширять позже. Куратор может попросить admin'а через Telegram.

### Q2. Нужен ли approval flow?

**Варианты:**
- A) Нет, admin меняет сразу
- B) Запрос от старого владельца → подтверждение admin'ом
- C) Двойное подтверждение (старый владелец + новый владелец)

**Рекомендация:** A для MVP. Для нестандартных случаев — admin принимает решение по обращению в поддержку.

### Q3. Что делать со старым QR-кодом?

**Варианты:**
- A) Старый QR остаётся валидным (тот же UUID билета) — просто меняется ФИО/email
- B) Старый QR инвалидируется, генерируется новый
- C) Старый QR на 24 часа остаётся валидным (grace period), потом новый

**Рекомендация:**
- Если меняется только ФИО/email/телефон (та же запись `tickets.id`) — **вариант A**, QR не меняется (он привязан к `ticket.id`).
- Если бизнес хочет «передать билет другу как новую сущность» — **вариант B** (старый билет помечен `revoked`, создан новый билет с новым UUID и новым QR в том же заказе). Так чище для аудита.

**Нужен ответ пользователя:** «смена владельца» = переименование владельца (вариант A) или полная передача билета как новой сущности (вариант B)?

### Q4. Уведомления

**Варианты:**
- A) Старому владельцу — «ваш билет передан»
- B) Новому владельцу — «вам передан билет, QR во вложении»
- C) Обоим (выбор admin'а в форме)
- D) Никаких автоматических уведомлений

**Рекомендация:** C (опции на чекбоксах в форме admin'а).

### Q5. Сохраняем ли историю всех владельцев?

**Ответ:** Да, обязательно — это основной смысл фичи. Каждая смена → событие `ticket.owner_changed` в `domain_history`.

### Q6. Что с гостевой анкетой?

**Варианты:**
- A) Старая анкета остаётся, новому владельцу не нужно её заполнять
- B) Старая анкета удаляется, новому владельцу шлётся ссылка на новую анкету
- C) Старая анкета помечается архивной, шлётся новая

**Рекомендация:** B (или C). Анкета привязана к человеку (имя, телефон, telegram), и если человек сменился — анкета должна быть от нового. Это влияет на:
- список гостей фестиваля (новое имя в реестре)
- статистику оргкомитета (правильное кол-во заполненных анкет)

### Q7. Сколько раз можно менять владельца?

**Варианты:**
- A) Неограниченно
- B) Не более N раз (например, 3) — против схем мошенничества
- C) Жёсткие правила «не позже чем за X дней до фестиваля»

**Рекомендация:** для MVP A (неограниченно), но в UI показывать счётчик смен — admin будет видеть подозрительные кейсы и сам решать.

### Q8. Что с ценой и промокодом?

**Ответ предполагается:** цена и промокод НЕ переносятся — это атрибуты заказа, не билета. Смена владельца не возвращает деньги предыдущему владельцу — это договорённость между ними, мы только меняем имя в системе.

**Нужно подтверждение** пользователя.

## 7. Оценка размера и привязка к релизу

### Размер: **L** (Large — 2-3 спринта)

Разбивка:
- **S1** (1 спринт): подключить `HasHistory` к `Ticket`, добавить базовые события (`created`, `qr_generated`, `email_sent`, `revoked`, `data_changed`), UI «история билета» для admin — **выход в v2.7.0 или v2.6.1**
- **S2** (1 спринт): смена владельца билета (полный flow + UI + уведомления + перегенерация PDF) — **выход в v2.7.0 или v2.8.0**
- **S3** (1 спринт): Baza ↔ Backend синхронизация событий сканирования (вариант 5: локальная история в Baza + батч-синк) — **выход в v2.8.0**

### Привязка к релизу

| Релиз | Включить |
|-------|----------|
| **v2.7.0** | S1 (история билета + базовые события) — синхронно с audit-флоу в v2.7.0 |
| **v2.8.0** | S2 + S3 (смена владельца + Baza sync) — параллельно с Laravel 11 part 1 |
| **v3.0.0** | Аналитика для оргкомитета (US-6), личный кабинет владельца |

**Альтернативный план**: если бизнес настаивает на смене владельца раньше — можно вынести S2 в **v2.6.0** как отдельную мини-фичу (без полной истории, только событие `ticket.owner_changed` в `domain_history` агрегата `order`). Это компромисс — не идеально архитектурно, но даёт быструю бизнес-ценность.

## 10. Технические заметки для tech-lead

> Это **не часть спецификации**, а подсказки для будущей разработки. Tech-lead уточнит/перепишет.

- **`Ticket` сейчас не имеет `HasHistory`** — нужно добавить `use HasHistory;` (как в `OrderTicket`). Pull events на стороне Application или в CommandHandler.
- **`TicketDto` пока не содержит много полей** — для события `ticket.owner_changed` нужно расширить или использовать VO `OwnerInfo`.
- **`setInBaza` сейчас в репозитории Backend** — событие `ticket.sent_to_baza` логично писать на уровне Application (CommandHandler) после успешного вызова `setInBaza` — иначе будет нарушено правило «БД только в репозитории» (см. `CLAUDE.md` §1).
- **Маппинг Baza-id → Backend-uuid** — нужен сервис `BazaTicketResolver`, который умеет по `(тип_таблицы, kilter)` отдать `ticket.uuid`. Без него событие `ticket.scanned_at_festival` некуда писать.
- **Перегенерация PDF при смене владельца** — `ProcessCreatingQRCode` уже есть, нужно его передёрнуть с новыми данными. Старый PDF — удалить из `storage/app/public/tickets/{id}.pdf` или переименовать с суффиксом `.archive.pdf`.
- **Существующий `ChangeTicket` Command** — частично покрывает `ticket.data_changed`. Можно расширить, чтобы при изменении полей `name`/`email`/`phone` поднимало событие `ticket.owner_changed` если установлен флаг (или сделать отдельный Command).
