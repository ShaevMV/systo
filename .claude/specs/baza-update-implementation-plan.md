# План реализации обновления Baza (смены / финансы / роли / браслеты-QR / PWA / интеграция)

> **Статус:** ЧЕРНОВИК на утверждение владельцем (2026-06-19). Коммит — только после «ок».
> **Входные документы:** [baza-shift-finance-update.md](baza-shift-finance-update.md) (требования + решения), [admin-panel-placement-decision.md](admin-panel-placement-decision.md) (архитектура §0 + маршрут §6), [BAZA.md](../docs/BAZA.md) (текущее устройство).
> **Маршрут:** строго по 8 шагам §6 decision-doc. Каждая фаза = набор небольших PR, не ломающих текущий вход.

## Рамки (твёрдые решения владельца)
- Baza работает **независимо от org**; финансы **целиком в Baza**, без синка аудита в org.
- Оперативный контур КПП — в Baza как **offline-first PWA** (своя session-auth), org AdminFront — только read-only витрина (в самом конце).
- Связь org↔Baza: **ingest-API** (org→Baza, fallback прямая БД) + **вебхук «билет прошёл»** (Baza→org). Без общего auth, без cross-schema JOIN.
- **5 ролей в рамках смены** (`change_user.role`): Администратор / Начальник смены / Билетёр / Комендант КПП / Охранник. Инвариант: у смены обязан быть главный.
- Порядок: сначала надёжный офлайн-контур → потом финансы → в конце read-витрина org.
- Security-фикс впуска **уже сделан** (PR #84).

## Карта фаз и критический путь

```
Ф0 локализация ассетов ─┬─► Ф2 роли (change_user) ─► Ф6 браслеты+парковки ─┐
(офлайн-предусловие)    │                                                   ├─► Ф7 финансы (после ГЕЙТА Q4)
Ф1 добивка auth входа ──┘   Ф3 ingest-API ─► Ф4 вебхук ─► Ф5 PWA-каркас ───┘            └─► Ф8 синк + read-витрина org
```
**Критический путь к «надёжному офлайн-входу на фесте»:** Ф0 → Ф5 (PWA + локальный SSL) → перенос горячих экранов. Финансы (Ф7) не на критическом пути входа и гейтованы Q4.

## Фаза 3 — Контракт интеграции org→Baza (Шаг 3). Fallback держит доставку ✅ ЗАВЕРШЕНА
**PR #108 `feat(baza): ingest-API приёма билета` + #109 `feat(org): маршрутизация доставки`** (2026-06-20)
- [x] (org) `ActorType::BAZA` + конфиг S2S-канала `services.baza_ingest` (env `BAZA_INGEST_URL`/`BAZA_INGEST_TOKEN`).
- [x] (Baza) `POST /api/baza/ingest/ticket` — middleware `baza.ingest` (`X-Baza-Token`, `hash_equals`, зеркало org `qr.ingest`), модуль `scr/Ingest/` (тонкий `IngestTicketApplication` + `IngestRepositoryInterface`/`InMemoryMySqlIngestRepository`, **БД только в репозитории**). Идемпотентно по естественному ключу: el→`uuid`, spisok→`ticket_uuid`, live→`kilter`, auto→`(order_id,auto)`. live — только UPDATE `el_ticket_id`; нет строки → `success:false` (org откатится). Контракт `{target, ticket{}}`.
- [x] (org) `DeliverTicketToBazaJob::deliver()` для каждой цели: `BazaIngestClient->send()` (ingest-API), при не-`true` — **текущая прямая запись `mysqlBaza`** (fallback). Канал default OFF (нет url+token) → поведение org не меняется.
- ~~Версионированный DTO билета (published language)~~ — отложено: контракт пока неявный `{target, ticket{}}` (поля = `toArrayForBaza/toArrayForSpisok`); явный версионированный DTO — когда q/Baza зафиксируют published-контракт.
- **Критерий выполнен:** билет доезжает в Baza через API; при выключенном/упавшем API — через прямую БД (fallback); идемпотентность по UUID. Тесты: Baza `IngestTicketApiTest` (9) + org `BazaIngestRoutingTest` (4). На стенде роут отвечает 401 без токена; активация канала = задать `BAZA_INGEST_TOKENS` (Baza) + `BAZA_INGEST_URL`/`BAZA_INGEST_TOKEN` (org) в .env обоих.
