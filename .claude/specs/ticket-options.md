# Спецификация: Опции к билетам

> **BREAKING CHANGE.** Перестраивается контракт `OrderTicket`: `ticket_type_id` и `promo_code` переезжают с уровня заказа на уровень гостя, добавляется массив `options[]`. Меняется фронт формы покупки, API создания заказов и схема БД. Дедлайн **2026-06-12** — scope, скорее всего, придётся резать (см. §11).

---

## 1. Бизнес-ценность

### Какую проблему решаем

Сейчас один заказ — один тип билета, один промокод на всех. Это упирается в три сценария:

1. **«Семья + друг»** — мама с ребёнком + взрослый друг хотят оформить одним чеком: ребёнку «Детский» (400₽), маме и другу — «Оргвзнос». Сейчас — **два отдельных заказа**: теряются конверсии, две транзакции в банке, два письма, жалобы в поддержку.
2. **Доп.товары/услуги к билету** (саженцы, мерч, парковка). Бизнес хочет продавать их **в один клик** при покупке: ожидаемый рост среднего чека +15–25%, продвижение экосистемной истории фестиваля, снижение барьера upsell.
3. **Промокод на конкретного гостя** — частая практика: организатор даёт промокод одному, тот оформляет на компанию. Сейчас скидка размазывается на весь заказ → искажается аналитика «сколько билетов по промокоду X».

### Целевая аудитория

| Роль | Что получает |
|------|--------------|
| **Гость** | Один заказ — разные типы билетов, опции, промокоды на конкретных людей |
| **Admin** | CRUD опций, разбивка заказа по гостям |
| **Pusher** | Friendly-заказ со смешанным составом + опции |
| **Curator** | Опции в заказах-списках — **под вопросом** (см. §9 Q5) |
| **Бизнес/маркетинг** | Каталог опций как новый продукт, точная аналитика по промокодам |

### Метрики успеха

| Метрика | До | Цель (30 дней) | Как измерить |
|---------|-----|----------------|--------------|
| Средний чек | ~3800₽ | +15% (≥4370₽) | AVG(`order.price.totalPrice`) |
| Доля заказов с >1 типом билета | 0% (невозможно) | ≥10% | COUNT distinct `ticket_type_id` в guests |
| Доля заказов с опциями | 0% | ≥20% от оргвзноса | COUNT(заказов с `order_ticket_options`) |
| Жалоб «не могу оформить семью одним заказом» | 5–10/мес | 0 | Telegram-поддержка |

### Что будет если НЕ сделать к 2026-06-12

- Старт продаж осеннего фестиваля идёт на старом контракте → теряется ~6 недель окна upsell-продаж
- Бизнес уже **анонсировал саженцы** организаторам — придётся откатить анонс, репутационный риск
- Конкуренты опережают по UX «корзина с разными билетами»

### Альтернативы (отбракованные)

| Вариант | Почему отбракован |
|---------|-------------------|
| **B1.** Только опции, без per-guest формата | Бизнес явно хочет per-guest (RESULTS.md). Половинчатое решение → через 2 месяца переделывать заново |
| **B2.** Двухэтапная миграция | Дедлайн один. Двойная миграция = двойной риск на проде |
| **B3.** Отложить до v2.7.0 | Невозможно — главный scope осеннего фестиваля |

## 2. User Stories

### US-1. Гость покупает смешанный состав

**Как гость**, я хочу в одной форме добавить себе оргвзнос, ребёнку — детский, машине — парковку, чтобы оплатить одной транзакцией.

**AC:** в `BuyTicket.vue` каждая карточка гостя имеет селект «Тип билета»; цена обновляется live; способ оплаты остаётся **один на заказ**.

### US-2. Гость добавляет опцию

**Как гость**, я хочу при покупке оргвзноса добавить «Саженец» за +200₽.

**AC:** под селектом ticket_type появляется список чекбоксов опций, привязанных к этому типу; live-пересчёт; API валидирует привязку (опция A не доступна для типа B).

### US-3. Гость применяет промокод на конкретного человека

**Как гость**, я хочу применить рефералку друга только к моему билету, а коллегу оплатить по полной цене.

**AC:** в карточке гостя — поле «Промокод» с кнопкой «Применить»; скидка считается только на этого гостя; лимит уменьшается на каждое применение (даже если применили дважды в одном заказе — `-2`).

### US-4. Админ управляет каталогом опций

**Как админ**, я хочу CRUD опций с привязкой к типам билетов.

**AC:** раздел «Опции» в админке; поля `name`, `price`, `ticket_type_ids[]`, `active`, `description?`, `image_url?`; при `active=false` скрывается из формы покупки, но не удаляется из уже оплаченных заказов; в списке видно кол-во применений.

### US-5. Pusher создаёт Friendly-заказ со смешанным составом

**Как pusher**, я хочу создавать Friendly-заказ с разными типами билетов и опциями.

**AC:** `/createFriendly` принимает новый формат `guests[]`; `price` теперь **считается автоматически** (не задаётся pusher руками — см. §9 Q10); pusher видит сводку перед подтверждением.

### US-6. Curator — заказы-списки

**Открытый вопрос (§9 Q5).** Рекомендация: НЕ добавлять опции в `createList` в v2.6.0 (списки бесплатные).

### US-7. Бизнес видит аналитику опций

**Как организатор**, я хочу отчёт «продажи опций за период».

**AC:** минимум — таблица «опция / кол-во / сумма» с фильтром по дате. Можно отложить в v2.7.0 если не успеваем.

## 3. Доменная модель

### Сущность `Option`

Новый модуль `Backend/src/Option/` (по аналогии с `Location`, `TicketTypePrice`).

```php
class OptionDto extends AbstractionEntity {
    Uuid    $id;
    string  $name;                  // "Саженец"
    float   $price;                 // 200.0
    ?string $description;
    ?string $image_url;
    bool    $active;                // default true
    Uuid    $festival_id;
}
```

### Связь Option ↔ TicketType — many-to-many через pivot

**Обоснование:** опция «Саженец» применима к нескольким типам («Оргвзнос», «Оргвзнос для регионов», «Оргвзнос мульти»); опция «Лесная карта» — к одному. Many-to-many покрывает оба кейса.

```sql
CREATE TABLE option_ticket_type (
  option_id        UUID NOT NULL,
  ticket_type_id   UUID NOT NULL,
  PRIMARY KEY (option_id, ticket_type_id),
  FOREIGN KEY (option_id)      REFERENCES options(id)     ON DELETE CASCADE,
  FOREIGN KEY (ticket_type_id) REFERENCES ticket_type(id) ON DELETE CASCADE
);
```

### Сущность `OrderTicketOption` (применённая опция, snapshot)

```sql
CREATE TABLE order_ticket_options (
  id                UUID PRIMARY KEY,
  order_ticket_id   UUID NOT NULL,           -- FK → order_tickets.id
  ticket_id         UUID NULL,               -- FK → tickets.id (заполняется при PAID)
  option_id         UUID NOT NULL,           -- FK → options.id
  option_name       VARCHAR(255) NOT NULL,   -- snapshot имени
  option_price      DECIMAL(10,2) NOT NULL,  -- snapshot цены
  quantity          INT NOT NULL DEFAULT 1,  -- кратность (см. §9 Q2)
  created_at        TIMESTAMP,
  updated_at        TIMESTAMP
);
```

**Почему snapshot:** если потом админ изменит цену/имя опции — историческая стоимость заказа должна сохраниться (так же как `order_ticket.price` фиксируется на момент создания, см. BUSINESS_RULES §4).

### Изменения в существующих DTO

**`GuestsDto`** (текущий — `Backend/src/Order/OrderTicket/Dto/OrderTicket/GuestsDto.php`):

| Новое поле | Тип | Описание |
|-----------|-----|----------|
| `ticket_type_id` | Uuid (NEW, required) | Тип билета этого гостя |
| `options` | Uuid[] (NEW) | Массив UUID выбранных опций |
| `promo_code` | ?string (NEW) | Промокод для этого гостя |
| `calculated_price` | PriceDto (NEW) | Расчёт на стороне backend, не доверяем фронту |

Существующие: `value`, `email`, `number`, `id`, `festivalId` — без изменений.

**`OrderTicketDto`:** удалить `ticket_type_id` и `promo_code` с уровня заказа. `priceDto` — агрегированная сумма по гостям.

### Domain Events

| Event | Изменения |
|-------|-----------|
| `ProcessCreateTicket` | Интерфейс тот же, внутри читает `ticket_type_id` из гостя, не из заказа |
| `ProcessOptionPurchased` (NEW) | Создаёт строки в `order_ticket_options` со snapshot. Вызывается внутри `ProcessCreateTicket` |
| `OrderOptionsChangedEvent` (NEW, history) | История изменения опций в `domain_history` |

Аналогично `/create`. `price` больше не задаётся pusher руками — считается автоматически (см. §9 Q10). `guests[].number` остаётся для live-ticket Friendly.

### POST `/api/v1/order/createList` — без изменений

Списки бесплатные. См. §9 Q5.

Префикс **`/api/v1/option`** (по аналогии с `/api/v1/location`):

| Метод | Маршрут | Middleware | Описание |
|-------|---------|------------|----------|
| POST | `/getList` | публичный | Фильтр: `name`, `festival_id`, `active`, `ticket_type_id` |
| GET | `/getItem/{id}` | публичный | Одна опция |

**`create/edit` Request:**
```json
{
  "data": {
    "name": "Саженец", "price": 200.0,
    "description": "...", "image_url": "...",
    "active": true, "festival_id": "uuid",
    "ticket_type_ids": ["uuid1", "uuid2"]
  }
}
```

### POST `/api/v1/order/calculatePrice` (NEW)

Публичный. Принимает тот же body что и `/create` (без `email/phone/city`). Возвращает разбивку:
```json
{
  "success": true,
  "guests": [{
    "ticket_type_name": "Оргвзнос", "base_price": 3800,
    "options": [{"name": "Саженец", "price": 200}],
    "subtotal_before_discount": 4000,
    "promo_code": "TMBR5X", "discount": 400, "total": 3600
  }],
  "order_total": 3600
}
```

Фронт вызывает на каждый change (debounced). На стороне `/create` тот же сервис вызывается повторно — **источник правды бэк**.

## 5. Расчёт цены — детально

### Алгоритм (псевдокод)

```
calculateOrderPrice(order):
  orderTotal = 0
  foreach (guest in order.guests):
    # 1. Базовая цена (с учётом волн)
    basePrice = TicketTypePriceRepo.getCurrentPrice(guest.ticket_type_id)

    # 2. Сумма опций (только тех, что привязаны к данному ticket_type)
    optionsPrice = 0
    foreach (optionId in guest.options):
      option = OptionRepo.getItem(optionId)
      if (option not in TicketType.options[guest.ticket_type_id]):
        throw ValidationError("Опция не привязана к типу билета")
      if (not option.active):
        throw ValidationError("Опция деактивирована")
      optionsPrice += option.price * quantity   # см. §9 Q2

    guestSubtotal = basePrice + optionsPrice

    # 3. Скидка
    discount = 0
    if (guest.promo_code):
      promo = PromoCodeRepo.find(guest.promo_code, guest.ticket_type_id, festival_id)
      if (promo == null): throw ValidationError("Промокод недействителен")
      if (promo.is_percent):
        # §9 Q7: процент от чего?
        discount = basePrice * (promo.discount / 100)    # рекомендация: только от билета
      else:
        discount = promo.discount                          # фиксированная сумма

    guest.calculated_price = {basePrice, optionsPrice, subtotal: guestSubtotal,
                              discount, total: guestSubtotal - discount}
    orderTotal += guest.calculated_price.total

  return orderTotal
```

### Сложные кейсы

| Кейс | Решение |
|------|---------|
| Процентный промокод — на (билет+опции) или только на билет? | **§9 Q7.** Рекомендация: **только на билет** (опции — допуслуга, скидка на «бесплатный саженец» бессмысленна) |
| Фиксированный промокод — на гостя или весь заказ? | **На гостя** (промокод теперь привязан к гостю) |
| Фиксированная цена билета (детский 400₽, групповой 24000₽) | Волны не применяются — `getCurrentPrice` для таких возвращает фиксированную цену. **Без изменений** |
| Лимит промокода исчерпан в момент покупки | Транзакционная проверка в `/create`. Если 5 гостей с одним промокодом, лимит 3 → **422**, не применяем частично |
| Дубликат промокода у двух гостей | Разрешено, каждое применение уменьшает лимит. См. §9 Q8 |
| Опция деактивирована между расчётом и оплатой | Snapshot цены в момент `/create`. После оплаты — не пересчитываем |
| Цена опции изменилась между расчётом и оплатой | Snapshot |
| `PriceDto` на уровне заказа | `sum(guests[].calculated_price.total)`. Поле `discount` = `sum(guests[].discount)` для отчётов |

## 6. Миграция существующих данных

### Что мигрируем

Таблица `order_tickets`:
- Колонки уровня заказа: `ticket_type_id`, `promo_code`
- JSON-колонка `ticket` — массив гостей (`[{value, email, ...}]`)

### Алгоритм

```
1. БЭКАП БД (5 копий, разные таймстампы, разные носители)
2. Миграции схемы (создание options, pivot, order_ticket_options)
3. Data migration (artisan-команда):
   foreach (order in order_tickets):
     foreach (guest in order.ticket JSON):
       guest.ticket_type_id   = order.ticket_type_id
       guest.promo_code       = order.promo_code
       guest.options          = []
       guest.calculated_price = распределить order.priceDto равномерно
     UPDATE order_tickets SET ticket = updated_json WHERE id = order.id
4. ВАЛИДАЦИЯ:
   - SELECT COUNT(*) — совпадает
   - SELECT SUM(price) — совпадает (до и после)
   - SELECT WHERE у гостя нет ticket_type_id — 0
5. ОСТАВИТЬ старые колонки order_tickets.ticket_type_id и promo_code 
   nullable до конца сезона осеннего фестиваля (страховка отката).
   Удалить в v2.7.0.
```

### Особые случаи

- **Friendly:** просто переносим `priceItem` каждому гостю, `discount = 0` если не было промокода.
- **Заказы-списки:** не имеют цены — миграция не нужна, поля и так NULL.
- **Внешние промокоды (`external_promo_codes`):** привязка к заказу сохраняется (на момент применения все гости были одного типа). Новые — к гостю.

### План отката

Если на проде критичный баг:
1. Откатить деплой бэкенда на v2.5.x.
2. Колонки `ticket_type_id`/`promo_code` всё ещё есть (мы их не удалили).
3. Лишние поля в JSON `ticket` (`ticket_type_id`, `options`, `promo_code`) — старая версия игнорирует.
4. **Известный риск:** новые заказы созданные в v2.6.0 со смешанными `ticket_type_id` — после отката будут отображаться как один тип (первого гостя). Митигация — короткое окно между релизом и откатом.

## 8. Edge cases / валидация

| Кейс | Поведение |
|------|-----------|
| Опция привязана к A, гость выбрал B + опцию A | **422** «Опция не доступна для этого типа билета» |
| Опция деактивирована, но передана в `options[]` | **422** «Опция деактивирована» (уже купленные не трогаем — snapshot) |
| `guests[]` пустой | **422** (как сейчас) |
| `guests[].ticket_type_id` отсутствует/null | **422** «Не указан тип билета для гостя N» |
| Гость хочет удалить опцию после оплаты | **Запрещено**. Только админ через расширенный `changeTicket` |
| Промокод исчерпал лимит между расчётом и оплатой | **422**. Транзакционная блокировка (SELECT...FOR UPDATE) |
| Один промокод применён к двум гостям в одном заказе | Разрешено (см. §5 и §9 Q8), уменьшает лимит на 2 |
| Дубликаты UUID одной опции в `options[]` | Трактуется как `quantity = 2` (если разрешена кратность §9 Q2). Иначе **422** |
| Гость без email, но с опцией | Разрешено (email опционален), опция привяжется к ticket_id после оплаты |
| Расчёт не сходится фронт vs бэк | **Backend — источник правды**. Заказ создаётся по бэк-расчёту, разница показывается в success-сообщении |
| Опция стоит 0₽ (бесплатный мерч) | Разрешено, увеличит кол-во применений в отчёте |
| Опция привязана к удалённому ticket_type | ON DELETE CASCADE в pivot чистит связь автоматически |
| Способ оплаты не поддерживает все типы билетов в заказе | **§9 Q11**. Текущее решение: все способы поддерживают |

### Конфликт с дедлайном — что резать

**14.5 > 9 доступных дней.** Рекомендации по сокращению scope:

1. **US-7 (аналитика опций)** → в v2.7.0 (экономит ~0.5 дня)
2. **Q9 (история промокодов)** → минимум `created_by_user_id` + событие, UI — в v2.7.0
3. **US-6 (опции в заказах-списках)** → исключить (см. Q5)
4. **Промокоды-агрегатор** (RESULTS.md §II) → отдельная ветка `feat/v2.6.0-promocode-aggregator`, scope урезать до минимума необходимого для опций (per-guest промокод + history). Расширение прав на роли — в v2.7.0.
5. **qr.spaceofjoy.ru SSO** (RESULTS.md §III) → отдельная ветка `feat/v2.6.0-qr-sso-passport`. Если разработчик один — выносить **в v2.7.0** (после фестиваля), даже если бизнес настаивал.

**Если разработчик один:** реалистично уложиться **только в опции + per-guest формат + миграция**. Промокоды-агрегатор и Passport SSO — в v2.7.0.

**Это требует обсуждения с пользователем до старта** — кто-то должен сказать «ок, режем scope».

### Риски

| Риск | Митигация |
|------|-----------|
| Сложная миграция данных сломает прод | 5 бэкапов, тест на копии прод-БД, окно отката за счёт старых колонок |
| Live-расчёт цены замедлит UI | Debounce 200ms + кэш на бэке + индексы на `options.festival_id` и pivot |
| Race condition при применении промокода с лимитом | SELECT...FOR UPDATE на строке промокода |
| Конфликты merge между 7 ветками | Интеграционная ветка + частые ребейзы, мержим по очереди (1→2→3→...) |
| qr.spaceofjoy.ru не успеет обновить интеграцию | Спецификация их разработчику в день старта ветки 4 (минимум за неделю до релиза) |
