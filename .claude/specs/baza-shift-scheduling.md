# Baza — плановое расписание смен, авто-открытие, уведомления, данные заказа на КПП

**Статус:** черновик плана (2026-06-21). Источник — запрос владельца в сессии PM 2026-06-21 (6 пунктов) + ответы на 4 уточняющих вопроса.
**Контекст:** org → admin-only, Baza = offline-first контроль входа на КПП (Ф5). Топология — **облако-мастер, узел КПП опционален** (истина = облако; телефон = тонкий клиент со снимком + очередью; синк по сотовой).
**Связанные:** `.claude/specs/baza-f5-pwa.md`, `.claude/specs/baza-shift-finance-update.md`, `.claude/specs/baza-f2-roles-rbac.md`, TD-43 (нет `schedule:run` в Baza), TD-46 (бэкенд-вклад Ф5), TD-47 (блокировка + wipe).

## 0. Эволюция прежнего решения

В `baza-shift-finance-update.md` §122 было: *«Достаточно открыть/закрыть смену, плановое vs фактическое время детально не считаем»*. **Это решение пересмотрено владельцем 2026-06-21:** теперь нужно **плановое расписание** (раздать смены на дни феста наперёд) + **авто-открытие по времени** + **уведомления составу**. Визуализация-сетка «точки КПП × смены» (§123) остаётся целевой — это и есть календарь.

Ключевой принцип: **план и факт — разные сущности.** Текущая таблица `changes` = ФАКТ (открыли/закрыли, счётчики впуска). Добавляем отдельный слой ПЛАНА (`shift_schedules`), не ломая флоу впуска. Связь план→факт — `changes.schedule_id`.

## 2. Модель данных (PR-A, фундамент)

Стиль Baza: `scr/` неймспейс `Baza\`, БД только в репозитории, bigIncrements id (как `changes`/`users`), без FK-каскадов (как `change_user`).

### Таблица `shift_schedules` (ПЛАН смены)
| Поле | Тип | Назначение |
|------|-----|------------|
| `id` | bigIncrements | PK |
| `festival_id` | string | фестиваль (текущая константа) |
| `kpp_point` | string nullable | точка КПП (для сетки «точки × смены») |
| `shift_date` | date | день смены |
| `planned_start` | dateTime | плановое начало |
| `planned_end` | dateTime nullable | плановое окончание |
| `name` | string nullable | метка смены (напр. «Ночная») |
| `status` | string | `planned` / `opened` / `cancelled` |
| `chief_id` | unsignedBigInteger nullable | плановый начальник смены |
| `created_at`/`updated_at` | timestamps | |

Индексы: `(festival_id, shift_date)`, `status`.

### Таблица `shift_schedule_user` (плановый состав) — зеркало `change_user`
| Поле | Тип |
|------|-----|
| `id` | bigIncrements |
| `schedule_id` | unsignedBigInteger (index) |
| `user_id` | unsignedBigInteger |
| `role` | string (`ShiftRole`) |
UNIQUE `(schedule_id, user_id)`.

### `changes` — связь факт→план
Добавить `changes.schedule_id` (unsignedBigInteger nullable) — открытая по плану смена ссылается на свой план. Текущий ручной флоу (без плана) продолжает работать (`schedule_id = null`).

### Модуль (CQRS, как `Changes`)
`Baza/scr/ShiftSchedule/` — Repository(Interface + InMemoryMySql), Applications через `InMemorySymfony{Command,Query}Bus`:
- `CreateSchedule` (день/интервал/точка/состав/начальник)
- `EditSchedule` / `CancelSchedule` (→ уведомления, PR-C)
- `ListSchedules` (фильтр festival/дата/точка — для сетки)
- `GetMySchedule(userId)` (#2) — смены, где userId в `shift_schedule_user`, дата >= сегодня

## 4. Авто-открытие и топология (PR-C)

- Команда `baza:auto-open-shifts` крутится на **ОБЛАКЕ** (боевой Baza), не на опциональном узле КПП. Истина = облако; открытая смена доезжает на телефоны синком.
- **Предусловие — TD-43:** в Baza сейчас НЕ запущен `schedule:run` (нет cron/supervisord; `QUEUE_CONNECTION=sync`; worker монтирует только Backend). Без него ни авто-открытие, ни дренаж Ф4 не работают. **Нужна инфра-задача devops:** поднять `schedule:run` (cron/supervisord-программа) на облачном Baza.
- Логика: смена `planned` + `now >= shift_date+planned_start` + не `cancelled` + ещё не `opened` → `SaveChange::save(состав, now, festival, chief)` + `schedule.status=opened` + `changes.schedule_id`. Идемпотентно (повторный тик не открывает дважды — гейт по `status`).
