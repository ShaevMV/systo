# Project Memory

Этот файл хранит важные решения, предпочтения и контекст, чтобы команда не забывала их между сессиями.

---

## ⚖️ ФУНДАМЕНТАЛЬНЫЕ ПРАВИЛА (обязательны для всех агентов)

### 1. Чистая многоуровневая архитектура
**Обращение к базе данных происходит ТОЛЬКО внутри репозитория.**
- QueryHandler, CommandHandler, контроллеры, сервисы — НЕ обращаются к моделям/БД напрямую.
- Все обращения к БД — через методы репозитория.
- Источник: Роберт Мартин — «Чистая архитектура», глава «Зависимости» (Dependency Rule).

### 2. Коммиты только после одобрения
**Без явной проверки кода и подтверждения от пользователя — коммит НЕ делаем.**
- Сначала показываешь изменения пользователю.
- Ждёшь явного одобрения.
- Только потом делаешь коммит.

### 3. Рекомендуемые книги для всех агентов
- **Роберт Мартин — «Чистая архитектура»**
- **Роберт Мартин — «Совершенный код»**

При объяснении архитектурных решений — указывать источник.

### 4. ГЛАВНОЕ ПРАВИЛО: смотреть как было реализовано ранее
**Перед написанием нового кода — изучи существующую реализацию.**
- Если новая реализация НЕ соответствует существующей — **СПРОСИ у пользователя**.
- Не изобретай паттерны — используй те, что уже есть в проекте.

## 🧠 Архитектурные решения

- **Анкетирование**: Используем поиск типа анкеты по `code` (slug), а не по `name` или ID.
- **Статусы заказов**: Строгая матрица переходов через `Shared/Domain/ValueObject/Status`.
- **CQRS**: Команды для изменения, Query для чтения. Shared Bus.
- **Типизация**: Frontend state должен совпадать с Backend DTO (snake_case).
- **Безопасность**: Секреты в `.env`, реальные ключи в git **запрещены**.

## 💻 Технические предпочтения

- **Frontend**: Новые компоненты строго на **Vue 3 Composition API (`<script setup>`)**.
- **Code Style**: Строгий PHP 8.2 (constructor promotion, strict types, typed properties).
- **TDD**: Разработчик пишет тест до/вместе с кодом. Блокировка коммита без тестов.
- **Комментарии**: PHPDoc для публичных методов обязателен. Inline — только для сложной логики («почему», а не «что»).
- **Покрытие тестов**: Минимум **80%** для Application слоя.

## 🎨 Предпочтения пользователя

- **Коммуникация**: Формат ответов — краткий список (A/B/C) для выбора.
- **Дислексия**: Внимательно проверять опечатки в названиях файлов, методов и переменных.
- **Коммиты**: НЕ добавлять `FrontEnd/src/main.js` без явного запроса.
- **Фидбек**: Если пользователь говорит «тут неудобно» — записать в этот файл.
- **Деплой**: Только пользователь даёт добро на релиз. Локальный тест → прод.
- **Устройства**: Мобильных пользователей больше, чем десктоп.
- **Известные UX проблемы:**
    - **Таблицы:** На мобильных не влезают в экран (нужно скроллить или скрывать).
    - **Формы:** Неудобно заполнять с телефона (мелкие поля, не те клавиатуры).
- **Дизайн:** Используем текущую Bootstrap вёрстку, если не сломана. SEO/Accessibility не нужны.
- **Тёмная тема**: Хотелось бы, но сейчас много других приоритетных проблем.
- **Фирменный стиль**: Есть, но пока не может предоставить.

## 🔐 Безопасность и данные

- **Хостинг:** VPS
- **Мониторинг:** Sentry (бесплатный тариф, 5000 сообщений/мес). Хватает для ДДОС, но не для глубокого анализа.
- **Разработка:** Пока один разработчик, но в будущем нужен способ безопасной передачи секретов.
- **Seeders:** Должны быть актуальными для тестирования.
- **Логирование:** Нужно логирование действий админов (смена статусов, удаление) — писать в лог.
- **GDPR/152-ФЗ:** Пока нет явных требований, но анкеты содержат перс. данные (телефон, email, Telegram).
- **Бэкапы:** ~2 раза в год перед фестивалями + пару раз во время продаж.
- **Известные проблемы:**
    - **Гонка контейнеров:** Воркер поднимается раньше БД → зацикливание (commit -> rollback -> disconnect).
    - **Место на сервере:** Бывали переполнения диска.

## 🧪 Тестирование и QA

- **Формат баг-репортов:** Формальный (Шаги воспроизведения, Ожидаемый, Фактический).
- **Staging сервер:** https://org.spaceofjoy.ru/ (для проверки мобильной версии и реальных сценариев).
- **Локальное тестирование:** org.tickets.loc (основная разработка).
- **Тестовые данные:** Есть дамп `systo.sql` с прода — использовать как референс для идеальных сидеров.
- **Тестовая БД:** Желательно отдельную (например `systo_test`), чтобы не ломать локальный дамп разработки.
- **Биллинг:** Альфа-банк не тестируем (проверяем только до редиректа на шлюз).
- **Приоритет:** Сначала починить существующие юнит-тесты (ошибки подключения/транзакций), новые писать только после этого.
